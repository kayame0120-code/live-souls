## 0. 準拠するプロジェクト規約
- 認証は **Fortify限定**(Breeze禁止)。登録は招待制。
- DBは **PostgreSQL** / デプロイ **Fly.io(nrt)**。
- 機密情報(FCログインID・パスワード・メール・電話・住所)は `encrypted` キャストで暗号化保存。
- **マルチテナント**(全テーブル `user_id` スコープ・他人のデータは403)。ただし `attendance_photos` は例外(規約0-6の例外②、閲覧はメンバー間共有・書込削除は投稿者のみPolicy)。
- 状態は **ENUM禁止**、`string` + バリデーションで表現。
- 申込/当選履歴・当選率は **pivot(attendance_identity)から集計**。手入力させない。
- 年齢は `birth_date` から自動計算(年齢カラムを持たない)。
- 参戦写真の保存先: ローカル開発=`local`ディスク / 本番=`s3`ディスク(Fly Tigris・S3互換、`PHOTO_DISK`環境変数で切替)。**既に対応済み、追加対応不要**(規約0-8)。

## 1. 概要と目的
複数のFC名義を持つヲタクが、名義管理・参戦記録・当落管理・現場ロジ・写真・セットリストまでを1本のアプリに集約するv2.0改修。既存実装(0.1)を土台に、ユーザーストーリー(現場手帳アプリ_ユーザーストーリー.docx)由来の新規機能を追加する。**既存で要件を満たしている項目は変更しない**(2章参照)。

## 4. ロジック・処理フロー

### 4.1 AI一括登録(公演情報・セットリスト共通)
1. ユーザーが公演情報/セットリスト情報を貼り付け
2. Laravelアプリが入力を受け付け、同一Flyマシン上のOllamaへHTTP送信
3. OllamaがテキストをJSON化して返却
4. Laravelが人間確認テーブル経由でJSON検証・確認画面表示(既存`events.import`パターン踏襲)
5. 確認後にDB登録

緩和策(`[既定]`・変更は要確認):
- キュー(queue worker)経由で実行し、Web側リクエストと資源を奪い合わない
- 軽量モデル(1B〜3B級)から開始し、実測でメモリ不足ならFly Machine分離に切替
- Ollama呼び出しはサービス層で抽象化し、環境変更をコード変更なしで吸収する

### 4.2 更新通知(ホーム画面)
既存`FcMembership::isInRenewalWindow()`を呼び出し、該当する名義をホームにカード表示する(`HomeService`に新規メソッド追加)。表示閾値は既存の`renewalWindowStart()`(期限月の前月2日)をそのまま使う。メール等の能動配信は行わない。

### 4.3 チケット確認通知
`HomeService::getPendingConfirmations()`と同様のパターンで、公演日が近づいた`planned`の参戦予定に対し「チケット確認はお済みですか?」を一言添える。表示のみ、ストレージ・アップロード機能は持たない。
