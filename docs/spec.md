## 0. プロジェクト規約

| 規約 | 状態 |
|---|---|
| 認証は Fortify。Breeze不使用。登録は招待制（公開登録OFF） | 確定 |
| DBは PostgreSQL / デプロイ Fly.io (nrt)。空Laravelは開通済み | 確定 |
| 機密情報（FCパスワード・電話・住所・login_id・email）は `encrypted` キャストで暗号化保存 | 確定 |
| 状態カラムは ENUM禁止。`string` + バリデーションで表現 | 確定 |
| マルチテナント：ドメインテーブルを `user_id` でスコープ。他人のデータは**404**（存在秘匿＝スコープで不可視にする。403だとデータの存在が相手に漏れるため）。**写真の削除・編集のみ403**（閲覧はメンバー共有のため権限判定で弾く） | 確定 |
| **マルチテナントの明示的例外は4つのみ**（下記） | 確定 |
| Places APIのレスポンスを永続キャッシュしない（ToS遵守）。DB保存は「ユーザーが確定した入力値」のみ | 確定 |
| 写真ストレージ：ローカル=localディスク / 本番=Fly Tigris（S3互換） | 既定 |

### マルチテナント例外（閉じた列挙・4つ）

1. **venues**（共有マスタ・全ユーザー読取／追加可、削除は紐づく参戦0件時のみ）
2. **events**（★v1.2新設・公演情報の共有マスタ。venuesと同型。全ユーザー読取／追加／編集可、削除は紐づく参戦0件時のみ。一括インポートも全ユーザー可）
3. **attendance_photos**（メンバー間共有＋タイル表示に要る最小限の参戦情報。書込・削除は投稿者本人のみ）
4. **公演名等サジェスト**（events参照は読取のみ）

> 管理者ロール（is_admin / Gate）は導入しない。招待制そのものを実質の認可とみなす。events は user_id を持たない全ユーザー共通マスタであり、名義（fc_memberships）とは扱いが異なる。

## 1. 概要と目的

複数のFC名義を持ち複数端末を使うヲタクが、バラバラなメモ帳管理を1つに畳むためのWebアプリ。中核価値は「いつ・どの名義で入って・どこの席だったか」を後から一目で引けること。

名義（FC会員情報＝ログイン情報を含む金庫）、参戦記録（主役）、公演情報（共有マスタ）、会場ナレッジ（遠征インフラの再利用＋メンバーで蓄積する見え方マッピング）を1本に統合する。参戦記録が主役で、抽選は「ライフサイクル管理」ではなく参戦に付く履歴メタデータとして扱う。当面は個人＋招待した友人数名で利用し、不特定多数には開かない。

## 2. ユーザーストーリーと受け入れ条件

| No | ストーリー | 受け入れ条件（YES/NO判定可） |
|---|---|---|
| S1 | 帰りの電車で記録したい | 公演を選ぶだけで日付・会場・アクセスが自動で入り、名義を選んで保存できタイムラインに反映される |
| S2 | 過去を「いつ・どの席」で引きたい | 年フィルタ切替でき、各レコードに日付・会場・座席・名義が常に出る |
| S3 | 名義をFC（グループ）単位で管理したい | グループ＝FCそのもの。タブ切替で該当名義だけ表示。追加・改名は自由、削除は配下名義0件のときだけ可 |
| S4 | FCログイン情報を安全に取り出したい | パスワードは伏字、コピーボタンでクリップボードへ（平文を画面に出さない） |
| S5 | どの名義で申込み何が当たったか | 公演ごとに申込名義が行で並び、各行に当選/落選/未発表が出る |
| S6 | 名義ごとにどの申込が当たったか見たい | 名義詳細で、その名義の各申込が当選/落選/未発表のいずれかで一覧表示される（手入力なし）。**当選率などの割合計算はしない**（当落が分かればよい） |
| S7 | 会場情報を遠征のたび再利用したい | 会場詳細に共有情報（住所・最寄駅・キャパ）と個人メモ（定宿・交通費）が両方出る |
| S8 | 友人に配りたい | 招待コードを発行でき、そのコード保持者だけが登録できる |
| S9 | 申込時点で記録し当落を待ちたい | 申込登録で公演（セレクト）＋申込名義（複数）を登録でき、当落画面に「未発表」で並ぶ。当選を付けると参戦予定に昇格 |
| S10 | 会場の「見え方」をメンバーで蓄積したい | 参戦記録に写真を添付でき、会場詳細で全メンバーの写真が座席情報つきで一覧できる |
| S11 | 公演情報を全員で持ち寄りたい（★改） | 誰でも公演一覧テキストを貼ると日付・会場・公演名に分解され、確認・修正して共有マスタへ一括登録できる |

> v1.1のS11（ユーザー個人のツアー一括登録）は廃止。一括インポートは共有マスタ（events）投入に置き換え、名義選択とは切り離す。

## 4. データ設計

```
identity_groups（＝FC・タブ） ─< fc_memberships（名義） >─ persons（名義人）
                            │  🔒 password / login_id / email      🔒 phone / address
venues（共有マスタ） ─< events（★共有マスタ・公演情報）
                              │
attendances（参戦・主役） ─ event_id ──> events   （★日付/会場/公演名はeventsから参照）
     │  │
     │  ├─< attendance_identity（pivot） >─ fc_memberships   result: pending/won/lost
     │  └─< attendance_photos（メンバー間共有）
     └─ venues ── venue_notes（個人メモ・user_id分離）
users ─< invitations
```

🔒=暗号化カラム（`encrypted`キャスト）。

### events（★新設・公演情報の共有マスタ）

| カラム | 型 / 備考 |
|---|---|
| event_name | 公演名（自由記述） |
| event_date | 公演日（date） |
| venue_id | FK→venues |
| — | user_id を持たない。全ユーザー共通マスタ。追加・編集は全ユーザー可、削除は紐づくattendances 0件時のみ |

### fc_memberships（名義）

| カラム | 型 / 備考 |
|---|---|
| artist_name | 担当アーティスト名（自由記述） |
| member_no | 会員番号（コピー対象） |
| login_id 🔒 | ログインID/メアド（コピー対象・非メール系ID界隈のため email とは別に維持） |
| **email 🔒** | ★追加。FC登録メールアドレス（コピー対象・暗号化） |
| password 🔒 | 伏字＋コピー。画面に平文描画しない |
| joined_on | date（1日固定保存）。更新期間の計算元 |
| oshi_color | 名義の担当色。プリセット11色セレクトで選択・hex文字列で保存・丸スウォッチ表示 |
| group_id / person_id | タブ所属 / 名義人 |

### persons（名義人）

| カラム | 型 / 備考 |
|---|---|
| name | 氏名 |
| birth_date | 生年月日。年齢はここから自動計算（年齢カラムを持たない） |
| phone 🔒 / address 🔒 | 電話 / 住所 |

### attendances（参戦・主役）

| カラム | 型 / 備考 |
|---|---|
| **event_id** | ★FK→events。従来の event_name / event_date / venue_id を置換。日付・公演名・会場・アクセスはeventsとvenuesから参照（参戦テーブルには持たない） |
| seat_block / row / number | 座席（全て任意）。参戦登録での手入力対象 |
| seat_raw | 表示用。上記3フィールドから自動合成・手動編集可（手動優先） |
| status | `applied/planned/attended/skipped`（ENUM不使用） |
| companion / memo | 同行者 / メモ |

> 会場（venue_id）は attendances から直接持たず events 経由で解決する。同一公演に複数人が参戦しても events 1件を共有する。

### attendance_photos（参戦写真・メンバー間共有）

| カラム | 型 / 備考 |
|---|---|
| attendance_id | FK→attendances |
| user_id | 投稿者。閲覧は全メンバー可（スコープ例外）。書込・削除は本人のみ |
| path / caption | ストレージパス / 任意キャプション |

制約：1参戦5枚 / 1枚10MB / jpeg・png・webp・heic受付 / 保存時にEXIF（GPS等）除去。**heicのデコードとEXIF除去には libheif 導入が前提**（未導入では安全に保存できないため受付不可。§8移行で導入する）。

### attendance_identity（pivot・申込／当落の関節）

| カラム | 型 / 備考 |
|---|---|
| attendance_id | FK→attendances |
| fc_membership_id | FK→fc_memberships |
| result | `pending/won/lost`（ENUM不使用）。**申込ごとの当落ステータス**。これが分かればよく、当選率などの割合計算はしない |
| ticket_count | 申込枚数（既定1・任意） |

## 5. 主要ロジック

- **参戦登録**：公演を検索付きセレクト（events部分一致・インクリメンタル絞り込み）で選択。選択すると日付・会場・アクセスが自動表示（読取専用）。手入力は座席3フィールドと写真のみ。名義は複数選択可、選択分だけpivot生成。公演がマスタに無いときは `/events/create` へ誘導。
- **公演日経過→参戦確認**：status=planned で event_date を過ぎた参戦は、ホームに「参戦した？」確認を表示。確定操作で attended に遷移。**自動遷移はしない**（当選したが不参加＝譲渡・体調不良を attended として集計に乗せないため）。後から手動で skipped も可。
- **当落ステータス表示**：名義詳細に、その名義の各申込を当選/落選/未発表（pivot.result）で一覧表示する。**当選率などの割合計算はしない**（当落が分かればよいという方針）。
- **参戦・申込の削除規則**：won（当選）が付いた参戦は**削除不可**（当選履歴の保全。「行かなかった」は status=skipped への変更で表現）。status=applied、または全pivotが pending/lost（won 0件）の参戦は**削除可**（確認ダイアログ）。削除時は pivot・写真レコードを cascade 削除し、**ストレージ上の写真実体も削除**する。**テスト必須**。
- **パスワード/emailコピー**：画面は `••••••••`。コピー時のみ復号平文を `navigator.clipboard` へ。DOMに平文を出さない。
- **招待→登録**：発行→`/register/{code}` で存在・未使用・未失効を検証→成功時に used_by/used_at を埋める。二重使用はDB一意制約で弾く。
- **更新期間の自動計算**：有効期限＝入会月1日+1年−1日（毎年同月日）／受付＝有効期限月の前月2日〜有効期限日。例：2022-10入会→毎年9/30期限・8/2〜9/30受付。2月入会は年跨ぎ（12/2〜1/31）。PHP計算・カラム追加なし。受付期間内は「更新受付中」バッジ。カウントダウン・通知はやらない。
- **申込登録と当選昇格**：`/lots/create` で参戦を status=applied ＋pivot=pending で生成。当落はセレクト（pending/won/lost）で更新。won が1件付いたら status=planned に自動昇格（全lostはappliedのまま）。誤訂正でwon0件に戻っても自動降格はしない（手動で戻す）。タイムラインはapplied非表示。
- **見え方マッピング**：会場詳細に全メンバーのattendance_photosをタイル表示（写真/座席/公演日/投稿者名）。削除は投稿者本人のみ。
- **公演の共有マスタ管理**：events は全ユーザーが追加・編集可。新規作成は必ず検索付きセレクトを経由し、見つからないときだけ「新規作成」が出る（venuesと同じ導線）。同一 venue_id × 同一 event_date で登録しようとしたら警告表示（ブロックはしない・昼夜2公演があるため）。削除は紐づくattendances 0件時のみ。
- **貼り付け一括インポート**：`/events/import` でテキストを行分解→日付・会場・公演名を候補抽出→確認テーブルで編集→共有マスタへ一括登録。空欄・余計な記号は解析時に除外。解析不能行も捨てず空欄で残す。全ユーザーアクセス可、名義選択とは無関係。
- **会場オートフィル**：会場新規時にPlaces APIで住所をオートフィル→ユーザー確認・修正して保存（確定値のみ保存）。失敗・キー未設定は手入力フォールバック、登録をブロックしない。

## 6. スコープ外・未決

### 実装しない（提案もしない）

- 座席図ビジュアル（会場座席座標データが必要・沼る）。見え方は写真蓄積（S10）で解く。
- 更新日数カウントダウン・通知（状態表示までがMVP）。
- ローカル完結型UI（サーバー保存・複数端末同期が前提）。
- 公演情報の外部取得（スクレイピング・FC自動ログイン）— 著作権・規約・名義停止リスク。貼り付けインポート＋サジェストで代替。
- スクショOCR取込 — v1.2以降据え置き（貼り付けインポートの運用実績を見て判断）。
- 管理者ロール（is_admin/Gate）— events を全ユーザー共有マスタにしたため不要。招待制が実質の認可。

### 未決事項

なし（v1.1のE1はA案で確定済み：グループ削除は配下名義0件時のみ。events/venues削除も同型で紐づく参戦0件時のみ）。
