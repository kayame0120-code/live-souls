## 0. 準拠するプロジェクト規約
- 認証は **Fortify限定**(Breeze禁止)。登録は招待制。**Fortify標準の2要素認証(TOTP)を有効化する**(v2.5・セキュリティ正本による)。
- DBは **PostgreSQL** / デプロイ **Fly.io(nrt)**。
- **機密情報の保護は2階層**(v2.5で改訂。旧「`encrypted`キャストで暗号化保存」を置き換え):
  - **E2E(クライアントサイド暗号化)**: `fc_memberships.member_no` / `login_id` / `password`。
    サーバー・管理者・DBからは復号不可。鍵階層はエンベロープ方式。詳細はセキュリティ正本
  - **標準暗号化(`encrypted`キャスト・APP_KEY方式)**: `persons.email` / `phone` / `address` / `birth_date`。従来どおり
- **マルチテナント**(全テーブル `user_id` スコープ・他人のデータは403)。ただし `attendance_photos` は例外(規約0-6の例外②、閲覧はメンバー間共有・書込削除は投稿者のみPolicy)。
- 状態は **ENUM禁止**、`string` + バリデーションで表現。
- 申込/当選履歴・当選率は **pivot(attendance_identity)から集計**。手入力させない。
- 年齢は `birth_date` から自動計算(年齢カラムを持たない)。
- 参戦写真の保存先: ローカル開発=`local`ディスク / 本番=`s3`ディスク(Fly Tigris・S3互換、`PHOTO_DISK`環境変数で切替)。実装済み。
- **LLM連携は`LlmService`インターフェースで抽象化する。実ドライバは`openai`のみ実装する**
  (v2.6で改訂。v2.1〜v2.5の「開発=ollama/本番=クラウド」は実測により撤回。理由・詳細は4.1)。
  **E2E対象フィールドの平文をLlmServiceに渡すことを禁止する**(v2.5・セキュリティ正本No.11)。
- **アプリの表示言語は日本語**(v2.5)。`APP_LOCALE=ja`とし、バリデーションメッセージ・属性名(会員番号・入会日等)を含む全ユーザー向け文言を日本語で表示する。導入方法(公式langファイル/`laravel-lang`パッケージ等)はCC裁量。英語メッセージが1つでもユーザーに露出したら不合格。

## 1. 概要と目的
複数のFC名義を持つヲタクが、名義管理・参戦記録・当落管理・現場ロジ・写真・セットリストまでを1本のアプリに集約するv2改修。既存実装(0.1)を土台に、モックで具体化されたユーザーストーリー由来の新規機能を追加する。
v2.5で実機操作フィードバックとセキュリティ設計を、v2.6でAI一括登録の入力実態(スクショ・X画像文化)を反映した。

### 3.1 申込締切・当落発表日の所在について(`tours`ではなく`events`にした理由)
片倉さんの指示は「`tours`テーブルのカラムに追加する」だったが、**`events`に追加する形で確定させた**。
理由：改修版`mockup.html`の当落詳細画面(`#scr-lot-detail`)で、同一ツアー「LUMIÈRE「Prism of Night」」配下の
`events`ごとに異なる締切/発表日が表示されている(京セラドーム大阪＝締切07-20・発表07-28、東京ドーム昼公演＝発表予定07-15、
横浜アリーナ＝発表済み06-20で結果確定)。1つの`tours`行にカラムを持たせると、同一ツアー内で会場ごとに違う締切を
表現できない(値が1つしか持てない)。締切/発表日は「どの日程(=`events`)に申し込むか」に紐づく情報のため、
`events`側に置くのが技術的に正しい。**「片倉さんの指示と実際に作ったモックの中身が矛盾している」ケースなので、
黙って指示通りにもせず、黙って上書きもせず、ここに明記する。** 単発公演(1ツアー=1公演)の場合は`events`1行に
締切が乗るだけなので、`tours`に置いた場合と見た目の結果は変わらない。この判断に異論があれば差し戻してほしい。

### 3.3 公演タブのグループ束ねと`idol_groups`の二役について【v2.5で新設】
`idol_groups`はv2.3時点では「担当メンバー選択のための親マスタ」だけの役割だったが、v2.5で
「ツアーの分類軸(`tours.idol_group_id`)」という第二の役割を持つ。この二役は同一の実在概念
(アイドルグループ)を指すため、テーブルを分けない。設計上の帰結：
- シーダー投入されたグループ(STARTO等)と、ユーザーが画面から追加したグループが同じテーブルに同居する
- 担当メンバーが未登録のグループ(ユーザー追加分)も、ツアー分類には使える。担当メンバー選択画面で
  そのグループを選ぶとメンバー0件になるが、その場合は色スウォッチによる手動選択に自然にフォールバックする
  (既存の「手動で変える」動線があるため追加実装は不要)
- グループの削除UIは**設けない**(共有マスタであり、他ユーザーのツアー・名義が参照している可能性があるため)。
  誤登録の統合・削除が必要になった場合は運用(管理者操作)で対応する(`[既定]`)

## 4. ロジック・処理フロー

### 4.1 AI一括登録(公演情報・セットリスト・当落締切の3系統共通)【v2.6で入力方式・アーキテクチャを改訂】
1. ユーザーが対象の**画像(主入力)またはテキスト(従入力)**を投入する
   - **画像**：公式サイトのスクショ・Xで流通する画像(メモ帳スクショのセトリ等)をアップロード。
     **1回あたり最大5枚**(`[既定]`・複数枚のスクショにまたがるスケジュールに対応)。
     対応形式は既存写真受付と同じ**jpeg/png/webp**(iOSスクショ=PNG・X保存画像=JPGのためこれで実用上足りる。heicは対象外のまま)
   - **テキスト**：貼り付け欄は残すが折りたたみ表示とし、画面上の主役は画像アップロードに置く。
     解析パイプラインは画像と共通(入力がテキストか画像かの違いのみ)
2. Laravelアプリが入力を受け付け、`LlmService`インターフェース経由でLLMへ送信(画像はマルチモーダル入力として渡す)
3. LLMが内容をJSON化して返却
4. **JSONをそのまま編集可能なHTMLフォームへ渡し確認画面を表示する(DBへの一時保存はしない)**。
   既存`EventImportParser`→`events.import.parse`/`events.import.store`と同じステートレス構造を
   セットリスト・当落締切の2系統にもそのまま適用する(3章参照)。
   **解析用画像は永続保存せず、解析完了後に一時ファイルを削除する**(`[既定]`・3章参照)
5. 確認画面で内容を修正・除外行のチェック解除・(当落締切のみ)対象`events`の手動選択を行い、送信でDB登録

**入力方式を「画像主・テキスト従」とした理由(v2.6)**：運用実態として、公演スケジュール・セトリは
スクショとXの画像で流通するのが大半であり、テキストでコピーできる形はむしろ少数。トークンコストも
スクショ1枚≒1,500トークン前後・1解析1円未満で、テキストとの差は意思決定に影響しない規模
(公式ページまるごとコピペはむしろ画像より重くなる)。テキスト欄は維持コストが実質ゼロのため
逃げ道として残す。

**確認画面の追加要素(v2.5・公演系統のみ)**：確認画面に**ツアーの所属グループ選択**を含める。
抽出したツアー名・入力内容に`idol_groups.name`と一致する文字列があれば自動で候補提示し、
無ければ手動選択(未分類のまま登録も可)。グループがマスタに無い場合は確認画面からグループ追加UI(2.2)へ誘導する。

**セキュリティ制約(v2.5・`[確定]`)**：`LlmService`へ送信してよいのは公演情報・セットリスト・当落締切の
入力(画像/テキスト)のみ。**E2E対象フィールド(`member_no`/`login_id`/`password`)の平文をいかなる形でも
LlmServiceに渡してはならない**(セキュリティ正本・基準No.11)。

**アーキテクチャ(`[確定]`・v2.6で改訂)**:
- **実ドライバはOpenAI API(`openai`)一本とする**。APIキー(`OPENAI_API_KEY`)は人間が`.env`にセット(残高チャージ済み)
- v2.1〜v2.5の「ローカル開発=`ollama`」は**実測により撤回**：ローカル環境のスペックでは
  出力が途中で切れる(1度で情報を取り切れない)・応答が遅い問題が発生し、開発用途としても不成立と判明した(片倉実測・2026-07-12)
- **`LlmService`インターフェースによる抽象化は維持する**(`[確定・仕様書著者判断]`)。理由：
  ①テストで実APIを叩かないためのフェイクドライバが必要(実装必須)、②将来のモデル/提供元乗り換えの保険。
  抽象化のコストは設計済みで、撤廃しても何も軽くならない。**Ollamaドライバの実装のみをスコープから除外する**
- 利用モデルはビジョン対応モデルとする(画像入力が主のため)。具体的なモデル名の選定・更新はCC裁量
  (`.env`または設定ファイルで差し替え可能にしておく)
- キュー(queue worker)経由で実行し、Web側リクエストと資源を奪い合わない(既定のまま維持)

### 4.3 チケット確認通知【実装済み】

### 4.4 events.start_time の保存方式【v2.1で追記・要合意事項の反映】
`events.start_time`は`time`型カラム。**Eloquentの`'datetime:H:i'`キャストは使用しない**(PostgreSQLでは`time`列に`Y-m-d H:i:s`形式を書き込もうとして本番INSERTが失敗する。`EventDuplicateKeyTest`のFAILで実証済み)。
**アクセサ/ミューテータ方式で確定**(`[確定]`)：取得時=Carbonへ変換、保存時=`H:i:s`文字列へ正規化。重複判定(`EventService::findDuplicates`)も`H:i:s`正規化後に比較する。

### 4.5 名義複製ロジック【v2.1で追記】
複製元の`person`を選択 → 複製先`identity_group`を選択 → 担当メンバー選択(グループ→メンバー→色自動反映、上書き可) → FC固有情報(会員番号・パスワード・ID・入会日等)を新規入力 → 保存時は既存`person`のIDを参照して新規`fc_membership`を作成(`persons`は重複させない)。
**v2.5注記**：FC固有情報のうちE2E対象3項目は、入力時にクライアント側で暗号化してから送信する(セキュリティ正本・基準No.11)。「都度新規入力」の仕様のためE2E化との矛盾は生じない(security_requirements v1.1で確認済み)。

### 4.6 AI当落締切一括登録フロー【v2.2で新設・v2.6で入力方式改訂】
公演情報(会場・日付・開演)と当落締切(締切日時・発表日)は、公式サイト上で**別ページに分かれていることが多い**ため、
公演一括登録とは別の専用入口を持つ(`[確定]`・2.2参照)。1回のコピペ/スクショで両方を漏れなく拾わせようとしない。
1. ユーザーが締切情報の**画像(主)またはテキスト(従)**を投入(複数会場分の締切一覧のスクショ等。4.1の入力仕様と共通)
2. `LlmService`が各行を「会場名・日付・締切日時・発表日」に構造化してJSON抽出
3. 確認画面で各行に対し、会場名+日付から一致する既存`events`を自動マッチして候補表示する。**一致しない行は対象`events`を手動選択させる**(該当がまだ登録されていなければ、先に公演一括登録を案内)
4. 送信で該当`events`の`application_deadline`/`announce_date`を更新
5. **抜け漏れは手入力で救う**：一括登録で拾いきれなかった/間違えた締切は、公演詳細または当落詳細の編集画面から直接手入力・修正できる(AI一括登録は必須動線ではなく時短手段の一つという位置づけ)
6. 締切超過の判定は**サーバー側**で行う(`now() >= application_deadline`)。端末時計に依存しない(`[既定]`)

### 4.7 セキュリティ実装の準拠先【v2.5で新設】
E2E(エンベロープ方式)・リカバリーキー・2FA・再認証・CSP・クリップボード・アクセスログの
要件と合否判定は、すべて`security_requirements_v1.1.md`/`security_criteria_v1.1.md`に従う。
本書はセキュリティ要件を**重複定義しない**(二重管理による食い違いを防ぐため)。
実装が基準を満たせない場合はQUESTIONS.mdへ隔離する(基準ファイル「使い方」の規定どおり)。

## 9. デプロイ計画(スコープ境界)【v2.3で新設・v2.5で改訂】
片倉方針(2026-07-11)：360度ビューを別ブランチに分離し、2段階デプロイとする。

```
main
 │
 ├─ 本書確定機能（360度ビュー以外・セキュリティ改修を含む）── Deploy 1（本番投入）
 │
 └─ 360度ビュー専用ブランチ（arena-view統合用）── Deploy 2（arena-view完成後）
```

- **Deploy 1(main)**：本書(v2.6)およびモックで`[確定]`とした機能一式(360度ビューを除く)。`venues.arena_view_key`カラム自体は実データが無くても実害がないためそのまま含める(`[確定]`)。マイグレーション/`/up`/テストスイートの確認ラインを通してから本番投入する(REPORT.md V1〜V3の方式を踏襲)。
- **セキュリティ改修(E2E化)はDeploy 1に含める**(`[確定・仕様書著者判断]`・v2.5)。理由：Deploy 1を`encrypted`キャストのまま先行させると、本番に実データ(FC認証情報)が平文相当の運用で蓄積され、後からE2Eへ移行する際に「各ユーザーのログイン時に順次再暗号化する」複雑なマイグレーションが必要になる。初回からE2Eで出す方がデータ移行コストが構造的に低い。**ただしE2E実装の工数次第でDeploy 1のリリース時期が後ろへずれるトレードオフがある**。リリース優先ならE2Eを切り出してDeploy 1.5とする選択肢もあるため、異論があれば差し戻すこと。
- **会場詳細の「360度ビュー」タブUI**（見え方マッピングとの内部タブ併存・プレースホルダー表示）：`arena_view_key`が空でも表示上の実害が無いため、**Deploy 1に含めてよいと判断する**(`[確定・仕様書著者判断]`。片倉からの直接指示ではないため、異論があれば差し戻すこと)。
- **Deploy 2(新規ブランチ)**：arena-viewエンジン側の会場実装が完成し、埋め込み方式(7章No.1)が決まった時点で着手。mainとは別ブランチで統合してから、そちらをデプロイする。
- ブランチ名・Git運用・CI/CDの具体手順は本書(アプリ仕様)の対象外。CC向けの作業指示(`cc_instructions`)側で定める(プロジェクトのカスタム指示による層分けを踏襲)。
