### マルチテナント例外（閉じた列挙・4つ）

1. **venues**（共有マスタ・全ユーザー読取／追加可、削除は紐づく参戦0件時のみ）

## 1. 概要と目的

複数のFC名義を持ち複数端末を使うヲタクが、バラバラなメモ帳管理を1つに畳むためのWebアプリ。中核価値は「いつ・どの名義で入って・どこの席だったか」を後から一目で引けること。

名義（FC会員情報＝ログイン情報を含む金庫）、参戦記録（主役）、公演情報（共有マスタ）、会場ナレッジ（遠征インフラの再利用＋メンバーで蓄積する見え方マッピング）を1本に統合する。参戦記録が主役で、抽選は「ライフサイクル管理」ではなく参戦に付く履歴メタデータとして扱う。当面は個人＋招待した友人数名で利用し、不特定多数には開かない。

## 2. ユーザーストーリーと受け入れ条件

| No | ストーリー | 受け入れ条件（YES/NO判定可） |
|---|---|---|

| S4 | FCログイン情報を安全に取り出したい | パスワードは伏字、コピーボタンでクリップボードへ（平文を画面に出さない） |

| S6 | 名義ごとにどの申込が当たったか見たい | 名義詳細で、その名義の各申込が当選/落選/未発表のいずれかで一覧表示される（手入力なし）。**当選率などの割合計算はしない**（当落が分かればよい） |
| S7 | 会場情報を遠征のたび再利用したい | 会場詳細に共有情報（住所・最寄駅・キャパ）と個人メモ（定宿・交通費）が両方出る |
| S8 | 友人に配りたい | 招待コードを発行でき、そのコード保持者だけが登録できる |
| S9 | 申込時点で記録し当落を待ちたい | 申込登録で公演（セレクト）＋申込名義（複数）を登録でき、当落画面に「未発表」で並ぶ。当選を付けると参戦予定に昇格 |
| S10 | 会場の「見え方」をメンバーで蓄積したい | 参戦記録に写真を添付でき、会場詳細で全メンバーの写真が座席情報つきで一覧できる |
| S11 | 公演情報を全員で持ち寄りたい（★改） | 誰でも公演一覧テキストを貼ると日付・会場・公演名に分解され、確認・修正して共有マスタへ一括登録できる |
| S12 | 座った席から会場を見回したい（★v1.3追加） | 360°対応会場では、会場詳細から専用画面に遷移しフルスクリーンで会場を回遊できる。参戦経由で開いた場合は自分の座席にピックが立つ（座席情報が引ける範囲で）。未対応会場では導線自体が出ない |

> v1.1のS11（ユーザー個人のツアー一括登録）は廃止。一括インポートは共有マスタ（events）投入に置き換え、名義選択とは切り離す。
> S12の位置づけ：v1.2まで「座席図ビジュアル」はスコープ外だった（§6参照）。独自ジオメトリ生成方式の確立により条件付きで解禁したもので、S10（実席からの写真マッピング）とは役割が異なる（S10＝実際の見え方の実写／S12＝会場全体の空間把握と自席の位置）。両者は会場詳細に併存する。

## 4. データ設計

```
identity_groups（＝FC・タブ） ─< fc_memberships（名義） >─ persons（名義人）
                            │  🔒 password / login_id / email      🔒 phone / address
venues（共有マスタ・arena_view_key） ─< events（★共有マスタ・event_date + start_time）
                              │
attendances（参戦・主役） ─ event_id ──> events   （★日付/開演/会場/公演名はeventsから参照）
     │  │
     │  ├─< attendance_identity（pivot） >─ fc_memberships   result: pending/won/lost
     │  └─< attendance_photos（メンバー間共有）
     └─ venues ── venue_notes（個人メモ・user_id分離）
users ─< invitations
```

🔒=暗号化カラム（`encrypted`キャスト）。

### events（★新設・公演情報の共有マスタ）

| カラム | 型 / 備考 |
|---|---|
| event_name | 公演名（自由記述） |
| event_date | 公演日（date） |
| **start_time** | ★v1.3追加。開演時間（`time` nullable）。**同日昼夜2公演を別レコードとして区別する識別子**。一括インポートで抽出、公演登録では任意入力 |
| venue_id | FK→venues |
| — | user_id を持たない。全ユーザー共通マスタ。追加・編集は全ユーザー可、削除は紐づくattendances 0件時のみ |

> **昼夜別レコードの含意（★v1.3の破壊的変更）**：公演の同定粒度が「会場×日」から**「会場×日×開演時間」**に変わる。同一会場・同一日でも開演時間が違えば別 event レコードとなり、別々の参戦・別々の申込として扱える（例：大阪城ホール 7/29 の 13:30公演 と 18:00公演）。start_time が NULL の event 同士は「同一会場・同一日で時間未設定」として重複警告の対象になる（§5「公演の共有マスタ管理」）。

> **開演の源は events.start_time に一本化する（★v1.3確定・QV13-1）**：開演時間は「公演（event）の属性」であり、参戦（attendance）には持たせない。参戦カード・次の現場・参戦詳細・参戦登録に出す開演は、参戦が指す `event.start_time` を参照して表示する（`attendances` は開演を保持しない＝§4 attendances のカラム定義に start_time は無い）。過渡的に実装へ残っている `attendances.start_time` / `attendances.open_time`（v1.1/v1.2の名残）は開演の源として使わない。カラムの物理削除は本バージョンの破壊的変更に含めない（§8で扱わない）＝表示上デプリケートするに留める。開場時間（open_time）は events に相当カラムを持たないため v1.3 では表示要件に含めない。

## 5. 主要ロジック

- **参戦登録**：公演を検索付きセレクト（events部分一致・インクリメンタル絞り込み）で選択。選択すると日付・会場・アクセスが自動表示（読取専用）。手入力は座席3フィールドと写真のみ。名義は複数選択可、選択分だけpivot生成。公演がマスタに無いときは `/events/create` へ誘導。
- **公演日経過→参戦確認**：status=planned で event_date を過ぎた参戦は、ホームに「参戦した？」確認を表示。確定操作で attended に遷移。**自動遷移はしない**（当選したが不参加＝譲渡・体調不良を attended として集計に乗せないため）。後から手動で skipped も可。
- **当落ステータス表示**：名義詳細に、その名義の各申込を当選/落選/未発表（pivot.result）で一覧表示する。**当選率などの割合計算はしない**（当落が分かればよいという方針）。
- **参戦・申込の削除規則**：won（当選）が付いた参戦は**削除不可**（当選履歴の保全。「行かなかった」は status=skipped への変更で表現）。status=applied、または全pivotが pending/lost（won 0件）の参戦は**削除可**（確認ダイアログ）。削除時は pivot・写真レコードを cascade 削除し、**ストレージ上の写真実体も削除**する。**テスト必須**。
- **パスワード/emailコピー**：画面は `••••••••`。コピー時のみ復号平文を `navigator.clipboard` へ。DOMに平文を出さない。
- **招待→登録**：発行→`/register/{code}` で存在・未使用・未失効を検証→成功時に used_by/used_at を埋める。二重使用はDB一意制約で弾く。
- **更新期間の自動計算**：有効期限＝入会月1日+1年−1日（毎年同月日）／受付＝有効期限月の前月2日〜有効期限日。例：2022-10入会→毎年9/30期限・8/2〜9/30受付。2月入会は年跨ぎ（12/2〜1/31）。PHP計算・カラム追加なし。受付期間内は「更新受付中」バッジ。カウントダウン・通知はやらない。
- **申込登録と当選昇格**：`/lots/create` で参戦を status=applied ＋pivot=pending で生成。当落はセレクト（pending/won/lost）で更新。won が1件付いたら status=planned に自動昇格（全lostはappliedのまま）。誤訂正でwon0件に戻っても自動降格はしない（手動で戻す）。タイムラインはapplied非表示。
- **見え方マッピング**：会場詳細に全メンバーのattendance_photosをタイル表示（写真/座席/公演日/投稿者名）。削除は投稿者本人のみ。
- **公演の共有マスタ管理**：events は全ユーザーが追加・編集可。新規作成は必ず検索付きセレクトを経由し、見つからないときだけ「新規作成」が出る（venuesと同じ導線）。**重複警告のキーは `venue_id × event_date × start_time`（★v1.3改）**。同一会場・同一日でも start_time が異なれば別公演として通す（昼夜2公演）。start_time が両方 NULL で venue_id×event_date が一致する場合のみ「同一公演の重複では？」と警告表示（ブロックはしない）。削除は紐づくattendances 0件時のみ。
- **貼り付け一括インポート**：`/events/import` でテキストを行分解→日付・会場・公演名・**開演時間**を候補抽出→確認テーブルで編集→共有マスタへ一括登録。全ユーザーアクセス可、名義選択とは無関係。解析仕様は下記「一括インポート解析」に確定記載する（実装済みパーサ `EventImportDemo.jsx` の `parse()` を正とし、PHPサービスクラスへ移植する。挙動を勝手に変えない）。

### 一括インポート解析（★v1.3確定・実装済みパーサを正とする）

貼り付けた公演ページテキストを、正規表現＋状態機械で解析する（AI・外部通信は使わない）。ロジックは `EventImportDemo.jsx` の `parse()` が正であり、node で 45件一致の機械検証済み。PHP移植時に**挙動を変更しない**こと（変更が必要なら QUESTIONS.md へ）。

| 工程 | 規則 | ステータス |
|---|---|---|
| 本文境界の切り出し | 最初に会場行 `[都道府県] 会場名`（`/^\[(.+?)\]\s*(.+)$/`）が現れた行を本文開始とする。以降で `＜ご注意＞` / `※` / `一覧へ戻る` / `SHARE` のいずれかが現れた行を本文終端とする。境界外（ナビ・フッター）は解析対象外＝ノイズ遮断 | 確定 |
| ツアー名の特定 | キーワード非依存。見出し（`CONCERT` / `公演情報` / `CONCERT/STAGE`）の直後にある、ノイズ辞書に無い・5文字以上の実質行をツアー名とする。命名が自由（`DOME LIVE 'VoyAGE'` / `LIVE TOUR` 等）でも構造位置で拾える。確認テーブルで全行に一括適用・編集可 | 確定 |
| 会場 | `[都道府県] 会場名` を会場として抽出。以降の日付・開演時間はこの会場に紐づく（次の会場行まで） | 確定 |
| 日付 | `YYYY.MM.DD（曜）` を `YYYY-MM-DD` へ正規化。同一行に開演時間が続く場合はそれも1公演として拾う | 確定 |
| **開演時間** | `HH:MM` 単独行、または日付行末尾の時刻を start_time として拾う。**同一日に複数の時刻があれば各々を別公演行にする**（昼夜。例 14:00 と 18:30） | 確定 |
| アクセス情報 | `アクセス情報` 行以降のテキストは access として保持（会場のアクセス補完に使える。events本体には入れない） | 確定 |
| 未解析行 | いずれにも当たらない行は `unknown` として**捨てず保持**し、確認テーブルに「未解析として保持された行」として表示する（ユーザーが救済・破棄を選べる） | 確定 |
| 会場名寄せ | 確認テーブル確定時、同名 venue 文字列は1会場に寄せて venue_id を採番・解決する（既存 venues と照合、無ければ新規作成候補） | 確定 |
| 重複検出 | 確定時に `venue名 × event_date × start_time` が重複する行を検出し警告表示（**ブロックしない**）。昼夜は start_time が違うので重複扱いにならない | 確定 |

> 確認テーブルは編集・行除外が可能。ユーザーが確定した値のみ events へ INSERT する（Places同様、解析結果の生データは永続化しない）。start_time 列があるため同日昼夜が別行として区別されたまま登録される。

- **360°ビューの起動と座席ピック（★v1.3追加）**：会場詳細の「360°で見る」から `/venues/{id}/arena-view` へ遷移（別画面・フルスクリーン）。`arena_view_key` が NULL の会場では導線を出さない。参戦経由で開いた場合は参戦の座席情報をクエリで受け取り、次の**3段フォールバック**でピックを配置する（照合の厳密さをユーザーに強制しない設計）。
  - **① 完全一致**：seat_block + seat_row + seat_number が arena-view 側の座席ID体系へ引ければ、その席にピックを立ててカメラを初期化する（基本経路）。
  - **② 部分一致**：block だけ引ければそのブロック代表点にピックを寄せる（列・番が空 or 表記不一致でも動く）。
  - **③ 座席情報なし**：seat が空、または参戦を経由しない到達（会場単体で開いた等）では**ピックを立てず**、会場だけを自由回遊させる。
  - seat_block と arena-view 側座席IDの対応づけ（①の実現手段：入力をブロックのセレクトに寄せるか／照合時に正規化するか）は実装時の詳細で、`docs/cc_instructions_v1.3.md` で確定する。ここでは「①が引けなければ②→③に安全に落ちる」ことだけを仕様とする。
- **会場オートフィル**：会場新規時にPlaces APIで住所をオートフィル→ユーザー確認・修正して保存（確定値のみ保存）。失敗・キー未設定は手入力フォールバック、登録をブロックしない。
