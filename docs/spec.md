## 0. プロジェクト規約

| 規約 | 状態 |

| DBは PostgreSQL / デプロイ Fly.io (nrt)。空Laravelは開通済み | 確定 |
| 機密情報（FCパスワード・電話・住所・login_id・email）は `encrypted` キャストで暗号化保存 | 確定 |

### マルチテナント例外（閉じた列挙・5つ）

1. **venues**（共有マスタ・全ユーザー読取／追加可、削除は紐づく参戦0件時のみ）
2. **events**（★v1.2新設・公演情報の共有マスタ。venuesと同型。全ユーザー読取／追加／編集可、削除は紐づく参戦0件時のみ。一括インポートも全ユーザー可）
3. **tours**（★v1.4新設・ツアーの共有マスタ。venues/eventsと同型。全ユーザー読取／追加／編集可、削除は紐づくevents 0件時のみ）
4. **attendance_photos**（メンバー間共有＋タイル表示に要る最小限の参戦情報。書込・削除は投稿者本人のみ）
5. **公演名等サジェスト**（events/tours参照は読取のみ）

> 管理者ロール（is_admin / Gate）は導入しない。招待制そのものを実質の認可とみなす。events / tours は user_id を持たない全ユーザー共通マスタであり、名義（fc_memberships）とは扱いが異なる。

## 1. 概要と目的

複数のFC名義を持ち複数端末を使うヲタクが、バラバラなメモ帳管理を1つに畳むためのWebアプリ。中核価値は「いつ・どの名義で入って・どこの席だったか」を後から一目で引けること。

名義（FC会員情報＝ログイン情報を含む金庫）、参戦記録（主役）、公演情報（共有マスタ）、会場ナレッジ（遠征インフラの再利用＋メンバーで蓄積する見え方マッピング）を1本に統合する。参戦記録が主役で、抽選は「ライフサイクル管理」ではなく参戦に付く履歴メタデータとして扱う。当面は個人＋招待した友人数名で利用し、不特定多数には開かない。

## 2. ユーザーストーリーと受け入れ条件

| No | ストーリー | 受け入れ条件（YES/NO判定可） |
|---|---|---|
| S1 | 帰りの電車で記録したい | 公演を選ぶだけで日付・会場・アクセスが自動で入り、名義を選んで保存できタイムラインに反映される |
| S2 | 過去を「いつ・どの席」で引きたい | 年フィルタ切替でき、各レコードに日付・会場・座席・名義が常に出る |
| S3 | 名義をFC（グループ）単位で管理したい | グループ＝FCそのもの。タブ切替で該当名義だけ表示。追加・改名は自由、削除は配下名義0件のときだけ可 |
| S4 | FCログイン情報を安全に取り出したい | パスワードは伏字、コピーボタンでクリップボードへ（平文を画面に出さない） |
| S5 | どの名義で申込み何が当たったか | 公演ごとに申込名義が行で並び、各行に当選/落選/未発表が出る |
| S6 | 名義ごとにどの申込が当たったか見たい | 名義詳細で、その名義の各申込が当選/落選/未発表のいずれかで一覧表示される（手入力なし）。**当選率などの割合計算はしない**（当落が分かればよい） |
| S7 | 会場情報を遠征のたび再利用したい | 会場詳細に共有情報（住所・最寄駅・キャパ）と個人メモ（定宿・交通費）が両方出る |
| S8 | 友人に配りたい | 招待コードを発行でき、そのコード保持者だけが登録できる |
| S9 | 申込時点で記録し当落を待ちたい | 申込登録で**ツアー→日程をセレクトボックスで選び**＋申込名義（複数）を登録でき、当落画面に「未発表」で並ぶ。当選を付けると参戦予定に昇格 |
| S10 | 会場の「見え方」をメンバーで蓄積したい | 参戦記録に写真を添付でき、会場詳細で全メンバーの写真が座席情報つきで一覧できる |
| S11 | 公演情報を全員で持ち寄りたい（★改） | 誰でも公演一覧テキストを貼ると**ツアー名・日付・会場・開演時間**に分解され、確認・修正して共有マスタへ一括登録できる。**ツアー名は既存tourと自動照合され、無ければ新規作成候補になる** |
| S12 | 座った席から会場を見回したい（v1.3） | 360°対応会場では、会場詳細から専用画面に遷移しフルスクリーンで会場を回遊できる。参戦経由で開いた場合は自分の座席にピックが立つ（座席情報が引ける範囲で）。未対応会場では導線自体が出ない |
| S13 | ツアー単位で公演・当落をまとめて見たい（★v1.4追加） | 公演一覧・当落一覧はいずれもツアーカードとして表示され、タップすると配下の全日程・全申込がまとまって見られる。単発公演も「1公演だけのツアー」として同じ導線で扱える |

> v1.1のS11（ユーザー個人のツアー一括登録）は廃止。一括インポートは共有マスタ（events）投入に置き換え、名義選択とは切り離す。
> S12の位置づけ：v1.2まで「座席図ビジュアル」はスコープ外だった（§6参照）。独自ジオメトリ生成方式の確立により条件付きで解禁したもので、S10（実席からの写真マッピング）とは役割が異なる（S10＝実際の見え方の実写／S12＝会場全体の空間把握と自席の位置）。両者は会場詳細に併存する。
> S13の位置づけ：v1.3までの events は「ツアー」という上位概念を持たず、公演名は自由記述の文字列だった（表記ゆれ・名寄せ不能）。tours を独立エンティティにしたことで、S11のツアー名解決・当落/公演画面の階層化が両方成立する。

> **「+ 公演を追加」ボタンの分割（v1.4）**：ツアー一覧（`/events`）には「＋ツアーを追加」、ツアー詳細（`/tours/{id}`）には「＋日程を追加」を分けて置く。ツアー新規作成は名前だけの軽い操作にし、日程の追加はツアー詳細に入ってから行う（mockup.html準拠）。

> **公演選択のカスケード化（★v1.5）**：v1.4までは events を横断する自由記述の検索付きセレクトで公演（日程）を選ばせていたが、**ツアー名・会場・日付・開演時間を含む長い文字列から都度検索する入力は負担が大きい**ため廃止する。`/attendances/create` と `/lots/create` はいずれも**①ツアーのセレクトボックス→②そのツアー配下の日程のセレクトボックス**の2段階選択に統一する。②の選択肢はツアー選択後に絞り込まれるため、横断検索より選択肢が常に少ない。公演がマスタに無いときの新規作成導線（ツアー詳細の「＋日程を追加」・ツアー一覧の「＋ツアーを追加」）は維持する。

### 担当色プリセット（11色）

セレクトで選択。DBは `oshi_color` に hex文字列を保存（既存の任意hex値もスウォッチ描画されるだけなので移行不要）。プリセットはコード上の設定配列で管理し、増減は配列編集のみ。

| 名称 | hex | 名称 | hex |
|---|---|---|---|
| 赤 | #E60033 | 青 | #1D6EC9 |
| オレンジ | #F6851F | 紫 | #8957A1 |
| 黄 | #F2C500 | ピンク | #EE87B4 |
| 黄緑 | #8FC31F | 白 | #FFFFFF（カード上は枠線必須） |
| 緑 | #00A960 | 黒 | #2B2B2B |
| 水色 | #00AFEC | | |

> hex値はグッズ・ペンライトの色味からの近似で、公式定義ではない。表示の当たり判定用であり厳密な色再現ではない。運用で色味を差し替える場合は配列のhexを直接編集する。

## 4. データ設計

```
identity_groups（＝FC・タブ） ─< fc_memberships（名義） >─ persons（名義人）
                            │  🔒 password / login_id / email      🔒 phone / address
tours（★v1.4新設・共有マスタ） ─< events（★共有マスタ・tour_id必須 + start_time）
                              │        │
venues（共有マスタ・arena_view_key） ─┘  │
                                        │
attendances（参戦・主役） ─ event_id ──┘   （★ツアー名/日付/開演/会場/公演名はeventsとtoursから参照）
     │  │
     │  ├─< attendance_identity（pivot） >─ fc_memberships   result: pending/won/lost
     │  └─< attendance_photos（メンバー間共有）
     └─ venues ── venue_notes（個人メモ・user_id分離）
users ─< invitations
```

🔒=暗号化カラム（`encrypted`キャスト）。

### tours（★v1.4新設・ツアーの共有マスタ）

| カラム | 型 / 備考 |
|---|---|
| name | ツアー名（自由記述。例「LUMIÈRE LIVE TOUR 2026「Prism of Night」」） |
| — | user_id を持たない。全ユーザー共通マスタ。追加・編集は全ユーザー可、削除は紐づくevents 0件時のみ |

> **単発公演もツアーとして登録する**（確定）。1回きりのライブでも「1公演だけのツアー」として tours に1件作る。ツアー無しの event は存在しない（events.tour_id は必須）。これにより画面・ロジックの分岐（ツアー有無で表示を変える等）が発生しない。

### events（公演情報の共有マスタ）

| カラム | 型 / 備考 |
|---|---|
| **tour_id** | ★v1.4追加・**必須**。FK→tours。単発公演も1公演だけのツアーとして紐づく |
| **event_label** | ★v1.4改名（旧 event_name）。`string` nullable。個々の日程を区別する任意ラベル（例「大阪 2日目」「追加公演 名古屋」「昼公演」）。**ツアー名の重複部分はここに書かない**（ツアー名は tours.name が持つ）。空なら日付・会場・開演時間だけで表示 |
| event_date | 公演日（date） |
| start_time | 開演時間（`time` nullable）。**同日昼夜2公演を別レコードとして区別する識別子**。一括インポートで抽出、公演登録では任意入力 |
| venue_id | FK→venues |
| — | user_id を持たない。全ユーザー共通マスタ。追加・編集は全ユーザー可、削除は紐づくattendances 0件時のみ |

> **表示の組み立て**：公演の見出しは `tours.name`（＋ `events.event_label` があれば結合。例「LUMIÈRE ARENA TOUR「Aurora」」+「大阪 2日目」）。event_label が空の日程はツアー名のみ＋日付・会場・開演で表示する。
>
> **昼夜別レコードの含意（v1.3からの継続）**：公演の同定粒度は「会場×日×開演時間」。同一会場・同一日でも開演時間が違えば別 event レコードとなり、別々の参戦・別々の申込として扱える（例：大阪城ホール 7/29 の 13:30公演 と 18:00公演）。start_time が NULL の event 同士は「同一会場・同一日で時間未設定」として重複警告の対象になる（§5「公演の共有マスタ管理」）。

| カラム | 型 / 備考 |
|---|---|
| artist_name | 担当アーティスト名（自由記述） |
| member_no | 会員番号（コピー対象） |
| login_id 🔒 | ログインID/メアド（コピー対象・非メール系ID界隈のため email とは別に維持） |
| **email 🔒** | ★追加。FC登録メールアドレス（コピー対象・暗号化） |
| password 🔒 | 伏字＋コピー。画面に平文描画しない |
| joined_on | date（1日固定保存）。更新期間の計算元 |
| oshi_color | 名義の担当色。プリセット11色セレクトで選択・hex文字列で保存・丸スウォッチ表示 |
| group_id / person_id | タブ所属 / 名義人 |

### persons（名義人）

| カラム | 型 / 備考 |

| name | 氏名 |
| birth_date | 生年月日。年齢はここから自動計算（年齢カラムを持たない） |
| phone 🔒 / address 🔒 | 電話 / 住所 |

### attendances（参戦・主役）

| カラム | 型 / 備考 |
|---|---|
| **event_id** | ★FK→events。日付・開演時間・会場・アクセスはeventsとvenuesから参照。**公演名（見出し）は events.tour_id 経由で tours.name を参照**し event_label と結合（参戦テーブルには持たない）。昼夜別公演は別 event_id を指すので、参戦も昼夜で別レコードになる |

| companion / memo | 同行者 / メモ |

### attendance_photos（参戦写真・メンバー間共有）

| カラム | 型 / 備考 |
|---|---|
| attendance_id | FK→attendances |
| user_id | 投稿者。閲覧は全メンバー可（スコープ例外）。書込・削除は本人のみ |
| path / caption | ストレージパス / 任意キャプション |

### attendance_identity（pivot・申込／当落の関節）

| カラム | 型 / 備考 |
|---|---|
| attendance_id | FK→attendances |
| fc_membership_id | FK→fc_memberships |
| result | `pending/won/lost`（ENUM不使用）。**申込ごとの当落ステータス**。これが分かればよく、当選率などの割合計算はしない |
| ticket_count | 申込枚数（既定1・任意） |

### venues（会場・共有マスタ）

| カラム | 型 / 備考 |
|---|---|
| name | 会場名 |
| address / nearest_station / capacity | 住所 / 最寄駅 / キャパ（Placesオートフィル＋ユーザー確定値） |
| **arena_view_key** | ★v1.3追加。`string` nullable。360°ビュー設定の識別子（例 `osakajo_hall`）。**非NULLの会場だけ会場詳細に360°導線が出る**。値は arena-view 側の会場設定（topology + 寸法）に対応。NULL＝未対応会場（導線を出さない） |

## 5. 主要ロジック

- **参戦登録**：**①ツアーをセレクトボックスで選択→②選んだツアー配下の日程（event）をセレクトボックスで選択**（カスケード。②の選択肢は①で絞り込まれ、会場・日付・開演時間をラベルに表示）。選択するとツアー名・日付・会場・アクセスが自動表示（読取専用）。手入力は座席3フィールドと写真のみ。名義は複数選択可、選択分だけpivot生成。公演がマスタに無いときは対象ツアーの `/tours/{id}/events/create` へ誘導（ツアー自体も無ければ `/tours/create` から）。
- **公演日経過→参戦確認**：status=planned で event_date を過ぎた参戦は、ホームに「参戦した？」確認を表示。確定操作で attended に遷移。**自動遷移はしない**（当選したが不参加＝譲渡・体調不良を attended として集計に乗せないため）。後から手動で skipped も可。
- **当落ステータス表示**：名義詳細に、その名義の各申込を当選/落選/未発表（pivot.result）で一覧表示する。**当選率などの割合計算はしない**（当落が分かればよいという方針）。
- **参戦・申込の削除規則**：won（当選）が付いた参戦は**削除不可**（当選履歴の保全。「行かなかった」は status=skipped への変更で表現）。status=applied、または全pivotが pending/lost（won 0件）の参戦は**削除可**（確認ダイアログ）。削除時は pivot・写真レコードを cascade 削除し、**ストレージ上の写真実体も削除**する。**テスト必須**。
- **公演名表示は tours 経由の参照のみ（★v1.5確定）**：参戦記録一覧・参戦詳細・ホーム・名義詳細など、公演名を表示する箇所は必ず `attendances.event_id → events.tour_id → tours.name`（＋event_label）を辿って組み立てる。**attendances・events のどちらにも公演名の自由記述コピーを持たせない**（denormalizeしない）。v1.4で events.event_name を廃止した時点で構造的には満たされているが、v1.5でこれを明示の原則として確定する。今後どのような表示・検索・エクスポート機能を追加する場合も、この参照経路を経由しない公演名の保持・表示は禁止。
- **パスワード/emailコピー**：画面は `••••••••`。コピー時のみ復号平文を `navigator.clipboard` へ。DOMに平文を出さない。
- **招待→登録**：発行→`/register/{code}` で存在・未使用・未失効を検証→成功時に used_by/used_at を埋める。二重使用はDB一意制約で弾く。
- **更新期間の自動計算**：有効期限＝入会月1日+1年−1日（毎年同月日）／受付＝有効期限月の前月2日〜有効期限日。例：2022-10入会→毎年9/30期限・8/2〜9/30受付。2月入会は年跨ぎ（12/2〜1/31）。PHP計算・カラム追加なし。受付期間内は「更新受付中」バッジ。カウントダウン・通知はやらない。
- **申込登録と当選昇格**：`/lots/create` は**ツアー→日程のカスケード選択＋申込名義（複数選択・1つ以上必須）を選ぶだけ**の最小フォーム。送信すると参戦を status=applied ＋pivot=pending で生成する（**当落ステータスは作成フォームでは選ばせない**）。当落の更新は当落詳細画面（`/lots/tours/{tour}`）のセレクト（pending/won/lost）で行う。won が1件付いたら status=planned に自動昇格（全lostはappliedのまま）。誤訂正でwon0件に戻っても自動降格はしない（手動で戻す）。タイムラインはapplied非表示。
- **見え方マッピング**：会場詳細に全メンバーのattendance_photosをタイル表示（写真/座席/公演日/投稿者名）。削除は投稿者本人のみ。
- **ツアーの共有マスタ管理（★v1.4新設）**：tours は全ユーザーが追加・編集可。新規作成は `/tours/create` でツアー名のみ入力し、作成後はそのツアー詳細（日程0件）へ遷移する。削除は紐づくevents 0件時のみ（events同型の削除規則）。重複防止の強制はしない（同名ツアーの作成をブロックしない）が、一括インポート・手動作成いずれも既存ツアーの検索を先に促す導線にする（venues/eventsと同じ「まず検索、無ければ新規作成」パターン）。
- **公演（日程）の共有マスタ管理**：events は全ユーザーが追加・編集可。新規作成は必ず対象ツアーを経由する（`/tours/{id}/events/create`）。**重複警告のキーは `venue_id × event_date × start_time`**（ツアーをまたいでも同じ会場・日・時間なら警告対象）。同一会場・同一日でも start_time が異なれば別公演として通す（昼夜2公演）。start_time が両方 NULL で venue_id×event_date が一致する場合のみ「同一公演の重複では？」と警告表示（ブロックはしない）。削除は紐づくattendances 0件時のみ。
- **当落画面のツアー単位グルーピング（★v1.4新設）**：`/lots` はツアーごとにカードをまとめ、そのツアー配下に「当落待り（pivot.result=pending を含む申込）」と「発表済（pending 0件）」があるかをサマリ表示する。タップで `/lots/tours/{id}` に遷移し、配下の全 event に対する申込を「当落待ち」「結果」で区分表示する。**新しいテーブルは不要**：`attendances.event_id → events.tour_id` を辿ってグルーピングするだけの表示ロジック。
- **貼り付け一括インポート**：`/events/import` でテキストを行分解→ツアー名・日付・会場・**開演時間**を候補抽出→確認テーブルで編集（**ツアー名の解決を含む**）→共有マスタへ一括登録。全ユーザーアクセス可、名義選択とは無関係。解析仕様は下記「一括インポート解析」に確定記載する（実装済みパーサ `EventImportDemo.jsx` の `parse()` を正とし、PHPサービスクラスへ移植する。挙動を勝手に変えない）。

### 一括インポート解析（実装済みパーサを正とする。★v1.4でツアー解決を追加）

貼り付けた公演ページテキストを、正規表現＋状態機械で解析する（AI・外部通信は使わない）。ロジックは `EventImportDemo.jsx` の `parse()` が正であり、node で 45件一致の機械検証済み。PHP移植時に**挙動を変更しない**こと（変更が必要なら QUESTIONS.md へ）。

| 工程 | 規則 | ステータス |
|---|---|---|
| 本文境界の切り出し | 最初に会場行 `[都道府県] 会場名`（`/^\[(.+?)\]\s*(.+)$/`）が現れた行を本文開始とする。以降で `＜ご注意＞` / `※` / `一覧へ戻る` / `SHARE` のいずれかが現れた行を本文終端とする。境界外（ナビ・フッター）は解析対象外＝ノイズ遮断 | 確定 |
| ツアー名の特定 | キーワード非依存。見出し（`CONCERT` / `公演情報` / `CONCERT/STAGE`）の直後にある、ノイズ辞書に無い・5文字以上の実質行をツアー名とする。命名が自由（`DOME LIVE 'VoyAGE'` / `LIVE TOUR` 等）でも構造位置で拾える。**parse() が返す `tour` はそのまま tours.name の解決候補になる**（下記「ツアー名の解決」）。確認テーブルで全行に一括適用・編集可 | 確定 |
| 会場 | `[都道府県] 会場名` を会場として抽出。以降の日付・開演時間はこの会場に紐づく（次の会場行まで） | 確定 |
| 日付 | `YYYY.MM.DD（曜）` を `YYYY-MM-DD` へ正規化。同一行に開演時間が続く場合はそれも1公演として拾う | 確定 |
| **開演時間** | `HH:MM` 単独行、または日付行末尾の時刻を start_time として拾う。**同一日に複数の時刻があれば各々を別公演行にする**（昼夜。例 14:00 と 18:30） | 確定 |
| アクセス情報 | `アクセス情報` 行以降のテキストは access として保持（会場のアクセス補完に使える。events本体には入れない） | 確定 |
| 未解析行 | いずれにも当たらない行は `unknown` として**捨てず保持**し、確認テーブルに「未解析として保持された行」として表示する（ユーザーが救済・破棄を選べる） | 確定 |
| 会場名寄せ | 確認テーブル確定時、同名 venue 文字列は1会場に寄せて venue_id を採番・解決する（既存 venues と照合、無ければ新規作成候補） | 確定 |
| **ツアー名の解決（★v1.4追加）** | 確認テーブル確定時、抽出した `tour` 文字列を**venue名寄せと同じパターン**で解決する：既存 tours を検索し一致すればそれに紐付け、無ければ新規作成候補として提示。1回の貼り付けで tour は1つ（parse() は全行に同一 tour を適用する仕様のため、確認テーブルのツアー名入力欄は1箇所・全行一括適用） | 確定 |
| 重複検出 | 確定時に `venue名 × event_date × start_time` が重複する行を検出し警告表示（**ブロックしない**）。昼夜は start_time が違うので重複扱いにならない。**ツアーをまたいだ重複も対象**（同じ会場・日・時間なら別ツアーの登録でも警告） | 確定 |

> 確認テーブルは編集・行除外が可能。ユーザーが確定した値のみ tours／events へ INSERT する（Places同様、解析結果の生データは永続化しない）。start_time 列があるため同日昼夜が別行として区別されたまま登録される。event_label は parse() が抽出しないため空のまま登録され、必要なら確認テーブルで手動入力する。

- **360°ビューの起動と座席ピック**：会場詳細の「360°で見る」から `/venues/{id}/arena-view` へ遷移（別画面・フルスクリーン）。`arena_view_key` が NULL の会場では導線を出さない。参戦経由で開いた場合は参戦の座席情報をクエリで受け取り、次の**3段フォールバック**でピックを配置する（照合の厳密さをユーザーに強制しない設計）。
  - **① 完全一致**：seat_block + seat_row + seat_number が arena-view 側の座席ID体系へ引ければ、その席にピックを立ててカメラを初期化する（基本経路）。
  - **② 部分一致**：block だけ引ければそのブロック代表点にピックを寄せる（列・番が空 or 表記不一致でも動く）。
  - **③ 座席情報なし**：seat が空、または参戦を経由しない到達（会場単体で開いた等）では**ピックを立てず**、会場だけを自由回遊させる。
  - seat_block と arena-view 側座席IDの対応づけ（①の実現手段：入力をブロックのセレクトに寄せるか／照合時に正規化するか）は実装時の詳細で、`docs/cc_instructions_v1.3.md` で確定する。ここでは「①が引けなければ②→③に安全に落ちる」ことだけを仕様とする。
- **会場オートフィル**：会場新規時にPlaces APIで住所をオートフィル→ユーザー確認・修正して保存（確定値のみ保存）。失敗・キー未設定は手入力フォールバック、登録をブロックしない。

## 6. スコープ外・未決

### 実装しない（提案もしない）

- ローカル完結型UI（サーバー保存・複数端末同期が前提）。

## 8. v1.3 → v1.4 移行（破壊的変更あり）

v1.4はtoursの新設とevents.tour_id必須化を含む破壊的変更。**event_name → event_label のリネームは値の意味が変わる**（旧値＝ツアー名込みの全文／新値＝日程だけの差分ラベル）ため、単純なカラムリネームではなく「新カラムを追加し空で初期化→旧カラムをDROP」の手順を踏む。目視照合は一切させず機械検証で担保する。

| 順 | 作業 |
|---|---|
| 1 | バックアップ：ローカルDBを日付付きダンプへ（マイグレーション前に必須） |
| 2 | `tours` テーブル作成（name、user_idなし） |
| 3 | **既存 events から tours を逆生成**：現行の `event_name`（リネーム前）の値ごとに tours を1件作成し、同じ event_name を持つ event 群は同一 tour に集約する（異なる event_name は別 tour ＝安全側。自動での統合・分割の推測はしない）。生成した tour_id を該当 events に埋める。**機械検証**：全 events に tour_id が埋まったことを確認 |
| 4 | `events.tour_id` を NOT NULL 制約に変更（手順3の検証通過後のみ実行） |
| 5 | `events.event_label`（string, nullable）を新規追加。**値は全て NULL で初期化**（旧 event_name の全文は既に tours.name 側に移っているため、差分ラベルとしての自動抽出はしない・安全側）。検証通過後に旧 `events.event_name` を DROP |
| 6 | 表示の張り替え：公演名を表示している全箇所（ホーム「次の現場」／参戦記録カード／参戦詳細／当落画面／公演一覧・詳細）を `tours.name`（＋event_labelがあれば結合）の組み立てに変更。**表示箇所を洗い出してチェックリスト化**し、漏れをテストで担保 |
| 7 | 画面追加：`/tours/create`・`/tours/{id}`（日程一覧）・`/tours/{id}/events/create`（旧`/events/create`を置換）・`/lots/tours/{tour}`（当落詳細）。`/events`（公演一覧）と `/lots`（当落一覧）はツアーカード一覧に差し替え |

| 9 | mockup.html 改訂（ツアー一覧／ツアー詳細／当落のツアー階層／「+ツアーを追加」「+日程を追加」のボタン分割・360°ビュー導線を統合） |
| 10 | §7「テスト化必須」を全件テストで担保（**tour逆生成の機械検証（3の突合）**・**tour_id NOT NULL制約後の登録経路が必ずtourを持つこと**・**表示の張り替え漏れがないこと（6のチェックリスト消化）**・**一括インポートのツアー名解決（既存マッチ／新規作成の両方）**・**当落画面のツアー単位グルーピングが既存pivotデータと一致すること**・既存回帰（v1.3までの全テスト）） |
