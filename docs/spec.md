## 0. アプリ固有の規約（CLAUDE.md の技術前提に上乗せする項目のみ）

CLAUDE.md 記載済みの前提（Laravel12/PHP8.2 / ローカルSQLite・本番Fly Postgres / ENUM禁止 / main直コミット禁止 等）は繰り返さない。本アプリ固有の規約は以下。

| No | 規約 | ステータス |
|---|---|---|
| 0-1 | 認証は **Fortify**。Breeze不使用。会員登録は**招待制**（`Features::registration()` を無効化し `/register/{code}` 経由のみ許可） | [確定] |
| 0-2 | 機密情報（FCパスワード・電話・住所・login_id）は Eloquent の **`encrypted` キャスト**で暗号化保存。ハッシュ化しない（本人が復号して使うため） | [確定] |
| 0-3 | **マルチテナント**。全ドメインテーブルを `user_id` でスコープ。他ユーザーのデータへのアクセスは403（グローバルスコープ or 明示的where必須） | [確定] |
| 0-4 | 生成元は自作 **laravel-starter**（Fortify招待制入り）。現場手帖はその初回適用先 | [確定] |
| 0-5 | 状態カラムは string + `in:...` バリデーション（CLAUDE.md のENUM禁止を本アプリのstatus/resultに適用） | [確定] |
| 0-6 | マルチテナント原則(0-3)の明示的例外は次の3つ**のみ**: ①venues（共有マスタ） ②attendance_photos（メンバー間共有。タイル表示に必要な最小限の参戦情報＝座席・公演日・投稿者名を含む） ③公演名サジェスト（メンバーの attendances から event_name・event_date・venue_id を**読み取りのみ**）。これら以外の他ユーザーデータ参照は403。写真の書込・削除は投稿者本人のみ | [確定] |
| 0-7 | Places API のレスポンスを永続キャッシュしない（ToS遵守）。DBに保存するのは「ユーザーが確認・確定した入力値」のみ | [確定] |
| 0-8 | 写真ストレージ: ローカル=localディスク / 本番=Fly Tigris（S3互換） | [既定] |

**DB二層の注意（CLAUDE.md「SQLite/PGで挙動が割れる書き方の禁止」の適用）**: 日付計算（年齢・更新期間）・集計（当選率）はDB固有関数に依存せずPHP側 or Eloquent標準機能で行う。生SQL・DB方言を使わない。

## 1. 概要と目的

複数のFC名義を持ち複数端末を使うヲタクが、**バラバラなメモ帳管理を1つに畳む**ためのWebアプリ。
中核価値は「いつ・どの名義で入って・どこの席だったか」を後から一目で引けること。
名義（FC会員情報＝ログイン情報を含む金庫）、参戦記録（主役）、会場ナレッジ（遠征インフラ再利用＋**メンバーで蓄積する見え方マッピング**）を1本に統合する。
当面は片倉個人＋招待した友人数名での利用。不特定多数には開かない。

## 5. ロジック・処理フロー

### 5-1. 参戦登録（S1・30秒導線）
入力: 日付・公演名・会場・座席（ブロック/列/番 + seat_raw）・名義（複数選択可）・status・写真（任意）
1. 会場名は既存venuesを部分一致サジェスト。ヒットなければ新規venue作成（created_by=自分）。
2. 公演名は自分＋メンバーの過去 attendances.event_name から部分一致サジェスト。
3. attendances作成（user_id=認証ユーザー）。
4. 選択名義ごとに attendance_identity 作成（result既定=pending）。
5. 写真があれば attendance_photos 作成（EXIF除去して保存）。
6. 参戦記録タイムラインへリダイレクト。

### 5-2. 年齢の自動計算
persons.birth_date から**PHP側で**計算（DB関数不使用）。誕生日未到来を考慮した満年齢。null なら非表示。

### 5-3. 当選率集計（S6）
名義詳細で該当 fc_membership の attendance_identity を集計:
申込数=該当pivot件数 / 当選数=result='won'件数 / 当選率=当選数÷申込数（**申込0件は「—」表示・ゼロ除算しない**）。

### 5-4. 招待→登録（S8）
1. `/invitations` で発行 → code生成、issued_by=自分。
2. `/register/{code}`: codeが存在・未使用・未失効なら登録フォーム、そうでなければ拒否。
3. 登録フォームに同意文言「投稿した参戦写真はメンバー間で共有されます」を表示。登録＝同意。
4. 登録成功時: invitationのused_by/used_atを埋め、users.invited_by=issued_by。

### 5-5. パスワード・コピー（S4）
画面には `••••••••`。コピーボタン押下で `navigator.clipboard.writeText()` に**復号済み平文**を渡す。平文はDOMのテキストとして描画しない。会員番号・login_idも同機構。

### 5-6. 更新期間の自動計算 ※v1.1新設 [確定]
```
有効期限 = joined_on の月の1日 + 1年 − 1日（以後毎年同月日）
更新受付 = 有効期限月の前月2日 〜 有効期限日

例: 2022-10入会 → 有効期限 毎年9/30、受付 8/2〜9/30
例: 11月入会   → 有効期限 毎年10/31、受付 9/2〜10/31
例: 2月入会    → 有効期限 毎年1/31、受付 12/2〜1/31（年跨ぎ）
```
- **PHP側で計算**（DB関数不使用）。カラム追加なし・モデルのアクセサのみ。
- 名義詳細に「有効期限」「更新受付期間」を表示。今日が受付期間内（境界日含む）なら「更新受付中」バッジ。
- joined_on が null なら更新期間欄ごと非表示（エラーにしない）。
- **日数カウントダウン・通知は実装しない**（§スコープ外）。

### 5-7. 申込登録と当選昇格（S9）※v1.1新設 [確定]
1. `/lots/create`: attendances作成（status=`applied`）＋選択名義ごとに pivot 作成（result=`pending`）。名義は1つ以上必須。
2. `/lots` で公演ごとの名義行の result を won/lost に更新できる。
3. result=`won` が1件でも付いたら attendance.status を `planned` に自動昇格 [既定]。全lostなら `applied` のまま。
   - 誤入力訂正で won が0件に戻っても**自動降格はしない**（status は編集画面から手動で戻せる）[既定]。
4. `/attendances`（タイムライン）は **status=applied を表示しない**。planned/attended/skipped のみ表示（skippedはグレー表現のまま）。

### 5-8. 座席の合成 ※v1.1新設 [既定]
- 主入力: seat_block / seat_row / seat_number（全て任意）。
- seat_raw は「{block} {row}列 {number}番」で自動合成（空要素はスキップ）。
- seat_raw が手動編集されたら、以後は構造化フィールドを変更しても自動合成で上書きしない（手動値優先）。

### 5-9. 見え方マッピング（S10）※v1.1新設 [確定]
- 会場詳細に、その会場に紐づく**全メンバーの** attendance_photos をタイル表示。
- 各タイル: 写真 / 座席（attendance.seat_raw）/ 公演日 / 投稿者名。
- 削除は投稿者本人のみ（他ユーザーの削除試行は403）。0枚時は「まだ写真がありません」。

### 5-10. 貼り付け一括インポート（S11）※v1.1新設 [既定]
1. `/lots/import` のテキスト欄に公演一覧を貼り付け → 行分解し、各行から日付・会場・公演名の候補を抽出。
2. 解析不能行も**捨てず空欄のまま**確認テーブルに出す（ユーザーが手で埋めるか、行を除外できる）。
3. 会場は既存venue部分一致 → なければ新規作成（5-1と同じ規則）。
4. 申込名義を選択して一括登録（各行が attendances: status=applied ＋ pivot: result=pending になる）。

### 5-11. 会場オートフィル（Places API）※v1.1新設 [既定]
- 会場新規作成時、名称で Places API（Text Search）を検索 → 住所をフォームにオートフィル。
- ユーザーが確認・修正して保存。**保存値はユーザー入力扱い**（APIレスポンスの永続保存はしない・規約0-7）。
- APIキーは環境変数。**API失敗・キー未設定時は手入力にフォールバックし、会場登録をブロックしない**。

## 7. エッジケース

CLAUDE.md 品質規約により、算出式の境界値・エッジケースは**必ずテスト化**する。

| ケース | 期待挙動 | ステータス |
|---|---|---|
| 会場サジェストで同名会場が複数 | 住所付きで候補表示し選択させる。無選択なら新規作成 | [既定] |
| 名義を選ばず参戦登録 | 許可（一般参戦）。pivotは作らない | [既定] |
| statusがskippedの参戦 | 積み上げ「今年の参戦」にカウントしない。タイムラインには表示（グレー表現） | [既定] |
| 同一公演に同一名義で複数口申込 | pivotを複数行許可（ticket_countで枚数区別も可） | [既定] |
| グループ削除時に配下名義が残存 | **削除拒否**。「先に名義を削除または移動してください」。**テスト化必須** | [確定] |
| 他ユーザーのvenue_note/attendanceへの直アクセス | 403（user_idスコープで弾く）。**テスト化必須** | [確定] |
| 他ユーザーの写真の削除・編集試行 | 403（閲覧は可・書込削除は投稿者のみ）。**テスト化必須** | [確定] |
| birth_dateがうるう年2/29 | 平年は3/1到来で加齢とみなす（標準的満年齢計算）。**テスト化** | [既定] |
| 1月入会の更新期間 | 有効期限12/31・受付11/2〜12/31（同年内）。**テスト化必須** | [確定] |
| 2月入会の更新期間 | 有効期限1/31・受付12/2〜1/31（**年跨ぎ**）。**テスト化必須** | [確定] |
| 更新受付の境界日 | 受付初日（2日）・期限日当日はどちらも「受付中」判定に**含む**。**テスト化必須** | [確定] |
| joined_on が null | 更新期間欄は非表示（エラーにしない） | [既定] |
| インポートで日付が解析できない行 | 空欄で確認テーブルに残す。**無言で捨てない** | [既定] |
| Places API失敗・キー未設定 | 手入力フォールバック。エラーで会場登録を止めない | [既定] |
| seat_raw手動編集後に構造化フィールド変更 | 自動合成で上書きしない（手動値優先） | [既定] |
| クリップボードAPI非対応環境 | フォールバックで一時textareaコピー。不可なら伏字解除トグルを最終手段 | [既定] |
| 招待コードの同時多重使用 | 最初のトランザクションのみ成功、以降「使用済み」。DB一意制約＋used_atで担保。**テスト化必須** | [確定] |

## 10. v1.0 → v1.1 移行（実装済みコードベースへの適用）

| fc_memberships.club_name | カラムDROP | 破棄してよい（グループ名に集約済みの想定。DROP前にQUESTIONS.mdへ現物値を記録） |

### 10-2. 手順（順序厳守）
1. **バックアップ**: ローカルSQLiteファイルをコピー（`database/database.sqlite` → 日付付き.bak）。マイグレーション実行前に必ず行う。
