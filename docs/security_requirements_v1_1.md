# 現場手帖 セキュリティ対策 確定事項 v1.1 (2026-07-12)

## 本書の位置づけ
本書はClaude(チャット)とのセキュリティ設計セッションで確定した**要件**を記録するものであり、
**具体的な実装方式(鍵導出アルゴリズム・DBスキーマ・APIエンドポイント設計・Fortifyとの統合方法等)は
本書の対象外**とする。現行コードベースを確認せずに書いているため、実装詳細への推測は意図的に避けた。

ただしv1.1では、**「実装方式を全面的にCCへ委ねると、基準を全部満たしていても壊れる」設計が
通ってしまうケース**が2件見つかったため、その2点に限り方式レベルまで踏み込んで確定させた
(§鍵階層・§XSS)。それ以外は従来どおり方式をCCに委ねる。

実装判断は `security_criteria_v1.1.md`(監督基準ファイル)の全項目を満たす形でCC側に委ねる。
本書と基準ファイルは対になっており、実装後は基準ファイルに照らして検品すること。

spec.mdとの関係：本書はspec.mdを補完するセキュリティ特化ドキュメントであり、spec.mdを置き換えない。
spec.md §0の「機密情報は`encrypted`キャストで暗号化保存」という記述は、下記の2階層化により
**部分的に古くなる**(E2E対象フィールドは`encrypted`キャストではなくE2E方式に変わる)。
spec.md本体の改訂は別セッションで行い、本書では扱わない。

## 背景
現場手帖はFC(ファンクラブ)会員情報を扱うアプリであり、漏洩時に以下2種の異なる実害を持つ:

1. **アカウント乗っ取りリスク**：FCの会員番号・パスワード・会員IDが漏洩すると、他人のFCアカウントを
   乗っ取られる実害が生じる
2. **信用毀損リスク**：氏名・住所・電話番号等、本人以外(招待された身内)の個人情報を預かる立場のため、
   漏洩はアプリの信頼性そのものを損なう

この2つはリスクの性質が異なるため、保護レベルも分けて設計する(下記スコープ参照)。

想定する脅威は主に「DBバックアップ・APP_KEY等の漏洩」「サーバー設定ミスによる情報流出」であり、
**稼働中サーバーへの完全な侵入(RCE)への耐性は本対応のスコープ外**とする。稼働中のサーバーが
攻撃者に完全掌握された場合、クライアントサイド暗号化(E2E)であっても配信JSの改ざん等により
突破されうる。これはWeb版E2Eの一般的な限界であり、本対応で解決しようとしない。

**ただしv1.1で脅威モデルを1点補正する**：RCEはスコープ外だが、**XSS(特にストアドXSS)はスコープ内**とする。
現場手帖はメモ・会場ノート・同行者名・写真キャプション等、ユーザー入力がメンバー間で共有表示される
画面を多数持つ。XSSが1つでも成立すると、E2E対象データは「ブラウザ内で復号された直後の平文」を
そのまま外部送信されるため、E2Eの防御価値がゼロになる。**Web版E2Eにおける現実的な最弱点はサーバー侵入ではなくXSS**
であり、これを未対策のままE2Eだけ実装するのは費用対効果が成立しない(§XSS)。

## 決定事項

### スコープの2階層化 (v1.0から変更なし)
| 階層 | 対象フィールド | 保護方式 |
|---|---|---|
| E2E(クライアントサイド暗号化・サーバー/管理者も復号不可) | `fc_memberships.member_no` / `login_id` / `password` | ユーザーのログインパスワードから導出した鍵でクライアント側のみ復号。サーバーは暗号文のみ保持・処理する |
| 標準暗号化(既存`encrypted`キャスト踏襲) | `persons.email` / `phone` / `address` | 既存の`encrypted`キャスト(APP_KEY方式)のまま。変更なし |

E2E対象は「アカウント乗っ取りに直結する3項目」に限定し、それ以外の個人情報は既存方式を維持する。
全項目を同列にE2E化すると実装コストが跳ね上がるため、被害の質で線引きした。

### 鍵階層：エンベロープ方式で確定 【v1.1で新設・方式レベルまで確定】
v1.0の「ログインパスワードから導出した鍵で復号」という記述だけを渡すと、
**パスワードから直接データ暗号鍵を導出する実装**が成立してしまう。この実装は次の致命的欠陥を持つ:

- **現場手帖のログインパスワードを変更した瞬間、既存のE2E対象データが全て復号不能になる**
- リカバリーキーによる復元も、同じ理由で構造的に組み込めない

したがって**エンベロープ(鍵のラッピング)方式で確定する**(`[確定]`):

```
        [ユーザーのログインパスワード]        [リカバリーキー]
                    │                              │
              KDF(Argon2id)                  KDF(Argon2id)
                    │                              │
              ラッピング鍵A                  ラッピング鍵B
                    │                              │
                    └──────┬───────────────────────┘
                           ▼
         どちらでも開ける同一のマスターキー(ランダム生成・ユーザー単位で1つ)
                           │
                           ▼
       member_no / login_id / password の暗号化・復号(全名義共通)
```

- **マスターキー**：ユーザー単位で1つ、ランダム生成する。**パスワードから導出しない**
- サーバーが保持するのは「ラッピング鍵Aで包んだマスターキー」と「ラッピング鍵Bで包んだマスターキー」の
  2つの暗号文のみ。マスターキー平文・パスワード平文・リカバリーキー平文はいずれも保存しない
- **パスワード変更時はマスターキーを包み直すだけ**でよく、データの再暗号化は不要
- リカバリーキーは「マスターキーの別包装」として自然に位置づけられる
- **鍵導出はメモリハードKDF(Argon2id等)を必須**とする。「実績のある標準ライブラリを使う」だけでは
  低イテレーションのPBKDF2も通ってしまい、オフライン総当たりへの耐性が確保できないため

### XSS対策：E2Eの前提条件として必須 【v1.1で新設】
E2E対象データの平文は、**復号された瞬間ブラウザのメモリ上に存在する**。XSSが成立すれば
その平文はそのまま外部送信できるため、XSS対策はE2Eの「オプション」ではなく**成立条件**である。

- **Content-Security-Policy(CSP)を設定し、インラインスクリプトの実行を禁止する**(`[確定]`)
  - 実装時は nonce または外部ファイル化で対応する。`docs/mockup.html`はインラインscriptを持つが、
    あれはモックであり本番Bladeの実装制約ではない
- **E2E対象フィールドの平文が、暗号化前にサーバーへ送信されないこと**(`[確定]`)
  - 基準v1.0のNo.1は「サーバー側経路から平文を*取得*できないか」しか見ておらず、
    「フォーム送信でサーバーに平文が*届いて*から暗号化する」実装を弾けない。
    暗号化は必ずクライアント側で完了させ、サーバーには暗号文だけを送る
  - リクエストログ・エラーログ・LLM連携(spec 4.1)のいずれにも、E2E対象の平文を通さない

### 追加防御層 【v1.1で新設】
E2Eは「DB・バックアップ・管理者」に対する防御であり、**セッションを奪われた場合には無力**
(認証済み本人として復号できてしまう)。したがってセッション自体の防御を厚くする。

| 対策 | 内容 | 位置づけ |
|---|---|---|
| 2要素認証(TOTP) | Fortify標準機能の2FAを有効化する。追加実装はほぼ不要 | `[確定]` |
| 機微情報表示前の再認証 | 名義詳細のログイン情報ブロックを開く前に、Fortifyの`password.confirm`でパスワード再入力を要求する(1Password等と同型)。放置端末・肩越し閲覧への対策 | `[確定]` |
| クリップボード自動クリア | コピー後一定時間(30〜60秒)で空文字上書き。ベストエフォートであり完全性は求めない | `[既定]`(実装コストが高いと判明した場合は要確認へ) |
| 暗号文取得APIのレート制限・アクセスログ | 「どの名義の暗号文が・いつ・どのセッションから取得されたか」をサーバー側に記録し、レート制限をかける。**復号内容はサーバーから見えなくても、異常な取得パターンは検知できる**。ログには暗号文本体・鍵・平文のいずれも残さない | `[確定]` |

### 認証済み本人からは見える (v1.0から変更なし)
E2Eは「サーバー・管理者・DBから見えない」ことを意味し、「誰からも見えない」ではない。
正しいログインパスワードで認証された本人は、自分のセッション内でE2E対象データを
復号・画面表示・コピーできる(ブラウザ内でその場復号する)。
※v1.1の「機微情報表示前の再認証」により、表示の直前にもう一段パスワード確認が入る。

### リカバリーキーの定義 (v1.1で鍵階層との関係を明記)
- 現場手帖のログインパスワードとは**独立した別個の秘密**とする
  (「忘れた時の救済手段」が「忘れているもの自体」と同一では機能しないため)
- **ユーザー単位で1つ**発行する。名義(fc_membership)ごとに個別発行はしない。
  1つのリカバリーキーで、そのユーザーが持つ全名義のE2E対象データを復元できる
  → **v1.1の鍵階層では、これは「マスターキーがユーザー単位で1つ」と同義**である
- 初回発行時(登録時、または初回FC情報保存時)に一度だけ提示し、本人がオフラインで保管する
- 通常のパスワードリセット(メールリンク等)だけではE2E対象データに再アクセスできない設計とする。
  リカバリーキーを提示できない場合、E2E対象データは**復旧不能**になる(合意済みのトレードオフ)
- **パスワード変更(本人が旧パスワードを知っている場合)では、データは復旧不能にならない**
  (ラッピング鍵Aを作り直してマスターキーを包み直すだけで済むため)。
  「パスワードリセット(旧パスワードを知らない)」と「パスワード変更(知っている)」を混同しないこと

### 明示的に対象外としたもの
- `persons.birth_date`(誕生日)はE2E対象に含めない。年齢自動計算(spec 0章・確定済み)を
  サーバー側で維持するため。誕生日は標準の`encrypted`キャストのまま
- 稼働中サーバーの完全侵入(RCE)に対する完全防御(理論上、Web版E2Eの限界として達成不能)
- **XSSは対象外ではない**(v1.1で脅威モデルを補正。§XSS参照)

## CCへの申し送り
- 具体的なアーキテクチャ(DBスキーマ、Fortifyとの統合方法等)はCC側の技術判断とする。
  **ただし鍵階層(エンベロープ方式)とXSS対策(CSP・平文をサーバーへ送らない)は
  本書で確定済みであり、CC裁量の対象外**とする(v1.1)
- 判断の際は `security_criteria_v1.1.md` の全項目を満たすこと。満たせない場合は
  独自解釈で進めず、QUESTIONS.mdへ隔離すること
- クライアント側の暗号処理は独自実装を禁止し、実績のある標準ライブラリ(libsodium.js等)を使用する。
  鍵導出は同ライブラリのメモリハードKDF(`crypto_pwhash` = Argon2id等)を使う
- 名義複製機能(spec 4.5)は「FC固有情報を都度新規入力」する仕様のため、E2E化しても矛盾は生じない
  (確認済み)
- **spec 4.1のAI一括登録(LlmService)にE2E対象の平文を渡さないこと**。LLM連携が扱うのは
  公演情報・セットリスト・当落締切のみであり、名義のログイン情報は一切通らない(v1.1で明記)

## v1.1で追加した項目の出自について
§鍵階層・§XSS・§追加防御層は、**片倉さんからの直接指示ではなくClaude(チャット)側からの提案を
片倉さんが承認したもの**である。特に以下は費用対効果の判断が入っているため、異論があれば差し戻すこと:
- クリップボード自動クリア(効果はベストエフォート・実装コスト中)
- 暗号文取得APIのアクセスログ(招待制・数名規模のアプリで運用上の実益があるか)
