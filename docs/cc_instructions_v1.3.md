## 1. マイグレーション [確定]

順序厳守。各ステップ後に `php artisan migrate` が通ること・既存テストが緑であることを確認してから次へ。

### 1-1. `events.start_time` 追加

```
Schema::table('events', fn(Blueprint $t) =>
    $t->time('start_time')->nullable()->after('event_date')
);
```

- 既存 events は `start_time = NULL` のまま（従来の「会場×日」粒度を保持）。**データ変換はしない**。
- Eventモデルの `$fillable` に `start_time` を追加。`$casts` に `'start_time' => 'datetime:H:i'`（表示整形用・保存はtime）。

### 1-2. `venues.arena_view_key` 追加

```
Schema::table('venues', fn(Blueprint $t) =>
    $t->string('arena_view_key')->nullable()->after('capacity')
);
```

- 全会場 `NULL` で開始。**この時点では360°導線はどこにも出ない**（正常）。
- Venueモデルの `$fillable` に追加。アクセサ `hasArenaView(): bool => !is_null($this->arena_view_key)` を用意。

> ⚠️ arena_view_key に**値を入れるのは器の実装が完了しテスト緑になった後**。器が動く前に値を入れると導線が出て未完成画面に飛ぶ。値の投入は §6 の最後。

## 2. A: 昼夜別レコードの同定 [確定]

- 公演の同定粒度が「会場×日」→**「会場×日×開演」**に変わる。
- 参戦（attendances）は event_id で events を指すため、**昼夜で別 event を指せば参戦も自然に別レコードになる**。attendances 側のスキーマ変更は不要。
- 表示：参戦カード・公演カード・次の現場・参戦詳細に**開演時間を表示**する（mockup.html の `.ev-time` 準拠）。event_date と併記。start_time が NULL の公演は開演を表示しない（「開演 --:--」等のプレースホルダを出さない）。

## 3. B: 一括インポート解析のPHP移植 [確定]

### 3-1. 原則

- **`EventImportDemo.jsx` の `parse()` が正**。node で SAMPLE 入力に対し45件一致を機械検証済み。
- PHPサービスクラス（例 `App\Services\EventImportParser`）へ**挙動を変えずに移植**する。正規表現・状態機械・ノイズ辞書・境界判定を1:1で写経する。
- 解析ロジックを「改善」しないこと。改善余地に気づいても QUESTIONS.md へ回し、本書の範囲では既存挙動を維持する。

### 3-2. 移植する規則（jsxと同一に保つ）

| 要素 | jsx側の定義 | PHP移植の注意 |
|---|---|---|
| 会場行 | `/^\[(.+?)\]\s*(.+)$/` | プリ処理の全角スペース・タブ正規化も含めて写す |
| 日付 | `/^(\d{4})\.(\d{2})\.(\d{2})（[^）]+）\s*(\d{1,2}:\d{2})?/` | 全角括弧に注意。`YYYY-MM-DD` へ正規化 |
| 時刻 | `/^(\d{1,2}:\d{2})$/` 単独行、日付行末の時刻 | **同一日に複数時刻→各々別公演行**（昼夜）。ここが start_time の源泉 |
| 本文境界 | 会場行初出〜`＜ご注意＞/※/一覧へ戻る/SHARE` | 境界外はノイズ遮断 |
| ツアー名 | 見出し直後・ノイズ辞書外・5文字以上の実質行 | キーワード非依存の構造位置ロジックを維持 |
| 未解析行 | `unknown` として保持 | 捨てない。確認画面に出す |
| ノイズ辞書 `NOISE` / 見出し `HEAD` | jsx の定数をそのまま | 定数を移植 |

### 3-3. 確認テーブル → 登録

- 確認テーブルは編集・行除外可（mockup準拠）。**ユーザー確定値のみ** events へ INSERT（解析の生データは永続化しない）。
- 会場名寄せ：同名 venue 文字列は1会場に寄せ、既存 venues と照合し無ければ新規作成候補。
- 重複検出：確定時に **`venue_id × event_date × start_time`** で重複を検出し**警告表示（ブロックしない）**。start_time が違えば昼夜として通す。

### 3-4. 検証（REPORT.md に出力を貼る）

- [ ] jsx の SAMPLE と同一入力を PHP パーサに通し、**件数・各行の date / start_time / venue が完全一致**することをテストで突合。
- [ ] 昼夜（同日複数時刻）が別行に分かれること。
- [ ] ノイズ行（TOP/SHARE/フッター）が遮断されること。
- [ ] 未解析行が捨てられず保持されること。

## 4. C: 360°ビュー画面と導線 [確定]

### 4-1. ルート

```
GET /venues/{venue}/arena-view   → ArenaViewController@show
```

- `venue->hasArenaView()` が false なら **404**（マルチテナントと同じ「存在秘匿」の思想。導線が出ない会場のURLを直打ちされても存在を漏らさない）。
- マルチテナントのスコープ対象外（venues は共有マスタ）。認証は要（招待制の内側）。

### 4-2. 画面

- **別画面・フルスクリーン**。会場詳細（`/venues/{id}`）の「360°で見る」から遷移。戻るで会場詳細へ。
- 中身は既存の単体HTML資産（`arena-view` プロジェクトの `venue_engine_template.html` 系）を Blade に組み込む。会場ごとの設定は `arena_view_key` で解決する（key→会場設定の対応表を持つ。対応表の実体は arena-view 側から移植）。
- mockup.html の `#scr-arena-view` がUIの見た目の正（フルスクリーンステージ・戻るボタン・HUD・自席ピック）。

### 4-3. 会場詳細の導線 [確定]

- 会場詳細に「360°で見る」ボタンを置く。**`venue->hasArenaView()` が true のときだけ描画**（false なら要素ごと出さない）。mockup.html の `.arena-view-btn` が正。
- **参戦詳細（`/attendances/{id}`）側は無改修**。参戦詳細の「この会場のビューを見る」は従来どおり会場詳細へ入る導線（360°の出し分けは会場詳細側に集約する＝判定を1箇所に）。

### 4-4. 座席ピックの受け渡し（3段フォールバック）[確定]

参戦経由で会場詳細→360°へ来た場合、その参戦の座席を渡してピックを立てる。参戦を経由しない到達（会場単体）では座席を渡さない＝③になる。

| 段 | 条件 | 挙動 |
|---|---|---|
| ① 完全一致 | seat_block＋seat_row＋seat_number が座席ID体系へ引ける | その席にピック・カメラ初期化 |
| ② 部分一致 | block だけ引ける（row/number が空 or 不一致） | ブロック代表点にピック |
| ③ 座席なし | seat が空 or 参戦を経由しない到達 | ピックを立てず会場だけ回遊 |

**①が失敗したら例外を投げず②→③へ落ちること**。表記ゆれ・空欄で500を出さない。ここはテストで担保（§7）。

### 4-5. V4の確定（seat_block ↔ 座席ID の対応づけ）[既定]

spec.md §6 の未決V4をここで**[既定]として確定**する（変更したい場合は要確認）:

- **方式：照合時に正規化（入力は自由文字列のまま）**。参戦登録のseat入力UIは会場によらず現状の自由入力を維持し、360°起動時に照合レイヤで正規化して座席IDを引く。
- 理由：入力UIを会場ごとに出し分ける（ブロックセレクト化）と、360°未対応会場と対応会場でフォームが変わり実装・保守が二重化する。片倉の「微細な正確性作業をユーザーに強制しない」方針とも、フォールバックが吸収する前提が噛む。
- 正規化の実体（全角半角・「アリーナ」接頭辞の除去・ブロック表記の寄せ）は会場設定側の対応表に持つ。**引けなければ②→③に落とす**のが大原則で、正規化を完璧にしようとしない（沼らせない）。
- この方式で不都合が出たら QUESTIONS.md へ。
