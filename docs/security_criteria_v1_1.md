# 現場手帖 セキュリティ監督基準
version: v1.1
updated: 2026-07-12
sources: セキュリティ設計セッション(2026-07-12)・security_requirements_v1.1.md

## 使い方
CC側の実装・Claude(チャット)側の検品、双方が本ファイルを判定基準として使う。
実装が本ファイルの全項目を満たさない場合、独自解釈で進めず「要確認」としてQUESTIONS.mdへ隔離すること。
判定条件はすべてYES/NOで答えられる形にしてある。「〜を意識する」のような基準はここには置かない。

**v1.1の要点**：v1.0の基準(No.1〜8)は「全項目を満たしていても壊れている実装」を通してしまう
穴が2つあった(パスワード変更でデータが死ぬ／XSSで平文が抜かれる)。No.9〜11がその塞ぎであり、
**No.9〜11はNo.1〜8と同格の必須基準**である。No.12〜16は防御層の追加。

## 検品時の注意(v1.1で追記)
- **No.1とNo.11は別物**。No.1は「サーバーから平文を*取り出せない*か」、No.11は「サーバーに平文が
  *届いていない*か」を見る。フォームで平文を送信し、サーバー側で暗号化してから保存する実装は
  No.1に合格しつつNo.11に違反する。両方を独立に確認すること
- **No.4とNo.9は別物**。No.4は「パスワード*リセット*(旧パスワードを知らない)ではデータを復元できない」、
  No.9は「パスワード*変更*(旧パスワードを知っている)ではデータを失わない」。
  片方を満たすためにもう片方を壊す実装が出やすいので、必ず両方をテストで確認する
- No.9・No.11は**コードを読まないと判定できない**(画面を見ても分からない)。CCには
  「鍵階層の図」と「E2E対象フィールドが通る全経路の一覧」を提出させ、それに対して判定する

## 思想メモ(基準化保留)
- 招待制・数名規模という運用実態に対し、No.16のアクセスログは過剰である可能性がある。
  「異常検知の実益があるか」は運用してみないと判断できないため、費用対効果は要観察
  (基準からは外していないが、CCが「コストが見合わない」と申告した場合はQUESTIONS.md経由で再検討する)
