## R3-A. 既存データのE2E移行を強制化（片倉承認済み・`[確定]`）

### 決定事項
「移行は任意（バナーのみ）」から**「ログイン時に強制」へ変更する**。理由：未移行のレガシー行が
無期限に残ることは、このアプリの最優先事項（名義情報を絶対にDBへ平文相当で残さない）に反する。
任意運用では招待メンバーがタップを忘れる限り古い保護のまま残り続け、リスクでしかない。

### 実装内容

1. **ログイン後、その名義に1件でも未移行（レガシー）行がある場合、移行が完了するまで
   名義関連の主要画面（一覧・詳細・登録・編集・複製）へのアクセスを強制的にブロックする。**
   - 遷移先は既存の移行フロー（`e2e-ui.js initMigrationBanner()/runMigration()`）をそのまま使ってよいが、
     「バナーが出るだけ」ではなく**専用の移行画面へリダイレクト**し、完了するまで他の画面に戻れないようにする
   - **例外**：ホーム画面・ログアウト・移行画面自体はブロック対象から外す（詰みを避けるため）
2. **移行完了後、旧暗号文（`encrypted`キャストで保存されていた元の値）は破棄する。**
   - 具体的には、移行APIが新しい`e2e:`形式の値を保存した**同一トランザクション内**で、
     元のカラムをNULLにするか、判別不能な値で上書きする
   - **これにより、移行完了後は`FcMembership::readProtectedField()`のレガシー復号パスが
     その行に対して二度と呼ばれなくなる**（読み出す旧暗号文が存在しないため）ことをコードで保証する
3. **1ユーザーが複数の招待メンバーと共存する場合の挙動を確認する。**
   - 本人の名義にレガシー行がある場合のみブロックする（他ユーザーの移行未完了状態で
     自分がブロックされることがないか、`user_id`スコープを確認する）

### 受け入れ条件
- レガシー行を持つユーザーでログインし、移行画面以外へアクセスしようとするとリダイレクトされる
  Featureテストの実行出力
- 移行完了後、DBの当該行を直接確認（tinker等）し、**旧暗号文カラムが判別不能な値になっている**ことの出力
- 移行完了後、`readProtectedField()`のレガシー復号分岐がその行に対して通らない（新形式`e2e:`のみが
  返る）ことをテストで証明
- 全ユーザーが移行済みの状態でログインした場合、ブロックが発生しないことのテスト

## R3-B. AI一括登録のキュー化（`[確定]`・spec 4.1準拠）

### 背景
spec 4.1は元々「キュー(queue worker)経由で実行」を要求している。v2.6で画像主・最大5枚に変更した結果、
前回report実測（テキスト2行=3.76秒）から、**画像5枚では60秒のHTTPタイムアウトに到達するリスクがある**と
判明した。同期呼び出しのまま出荷することを許可しない。

### 実装内容
1. 公演・セットリスト・当落締切の3系統すべてで、LLM呼び出しを**同期実行からキュー実行へ切り替える**
   - 既に残置されている `ParseEventsWithLlm` job を土台にしてよい。セトリ・締切系統も同型のjobを用意する
2. **ユーザー体験**：画像/テキストを送信 → 「解析中です」の待機画面（ポーリングまたは同等の進捗表示） →
   解析完了で自動的に確認画面へ遷移
3. **失敗時**：ジョブが失敗した場合（タイムアウト・LLM側エラー・不正なレスポンス等）、
   日本語のエラーメッセージを表示し、**再実行を促す**（spec 6章の「勝手に補完しない」方針を維持）
4. ジョブの`failed()`は、R2-2で確認した通りペイロード全体をログしないこと
   （画像/テキスト本体や、万一混入した場合のE2E対象値をログに出さない）

### 受け入れ条件
- 画像5枚を投入し、同期リクエストが即座に返る（キューに積まれて待機画面が表示される）ことをテストで証明
- ジョブが実行され、完了後に確認画面へ到達するテスト（FakeLlmServiceで擬似遅延を入れてもよい）
- ジョブ失敗時に日本語エラーが表示されるテスト
- 実際に画像5枚（本物のOpenAI呼び出し、またはそれに近い遅延を模したテスト環境）で
  **リクエスト自体が数秒以内に返ってくる**ことを実測で示す（重い処理はジョブ側で行われている証明）

## R3-C. レガシー復号関数の証拠提出（未回答・再要求）

`FcMembership::readProtectedField()`（レガシー`Crypt::decryptString`経路）について、
前々回から2回にわたり求めている証拠が**まだ提出されていない**。R3-Aの実装後は移行が強制されるため
実害は縮小するが、**移行完了までの間、この関数が正しい条件下でのみ呼ばれることの証明は必要**。

以下4点を、コードの該当箇所＋テスト出力で示すこと。

1. **`password.confirm`必須か**：この復号を経由する画面・APIが、E2E対象データの表示と同格の
   再認証ゲートを通っているか
2. **移行フロー専用か**：この関数が、移行処理以外の経路（通常の一覧表示・API等）から
   呼ばれていないか。呼ばれている箇所を全てgrepで洗い出す
3. **移行後は呼ばれなくなるか**：R3-A実装後、移行完了行に対してこの関数が二度と呼ばれないことの証明
   （R3-Aの受け入れ条件と重複する部分はそちらの出力を参照でよい）
4. **レスポンスがログに乗らないか**：R2-2で洗った7経路の対象に、この復号のレスポンスが
   含まれていることを明示的なテストで示す（現状のR2-2テストが対象にしていたか不明なため）

### 受け入れ条件
- 上記1〜4それぞれに、コード該当箇所（ファイルパス:行番号）とテスト出力が付いている
- いずれかが満たせない場合、その場で仕様を直すのではなく`QUESTIONS.md`へ隔離し、次の判断を仰ぐこと
