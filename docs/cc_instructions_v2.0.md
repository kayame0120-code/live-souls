# CC作業指示書 v2.0 (2026-07-10)

## 参照仕様
本書は `docs/spec_v2.0.md` の実装順序・境界線を指示するものであり、仕様そのもの(何を作るか)は spec_v2.0.md が正本。本書とspec_v2.0.mdが矛盾する場合はspec_v2.0.mdを優先し、矛盾自体をQUESTIONS.mdに記録すること。
恒久ルール(自律ループ・検証ライン・裁量境界・デプロイ境界)は `CLAUDE.md` に従う。本書はアプリ固有の作業順序のみを扱う。

## 大原則
- `[未決]` とマークされた項目は実装しない。QUESTIONS.mdに積んで片倉の判断を待つ。埋め方を推測しない。
- 動いている既存ロジックを壊してから作り直さない。新パイプラインが動作確認できるまで、廃止対象(EventImportParser)は残す。
- インフラ変更(Ollama同居)は証拠優先で段階的に進める。一気に本番投入しない(詳細は5章)。
- 1タスク1PR相当。複数エピックを1つの変更にまとめない。

## 5. AI一括登録基盤(Ollama) — 段階的に進める

Ollamaは本番Flyマシンに組み込む方針が確定しているが、リソース競合リスクを片倉さんが認識のうえで許容した決定であり、**一気に本番投入しない**。evidence-first-opsの原則(証拠→原因→合意→解決)に沿って、以下の順序で進めること。

1. **前例確認**: 現在のFlyマシンのスペック(`fly.toml`・`fly status`・現行のメモリ使用量)を確認する。推測で「足りるはず」と判断しない
2. **実測**: 候補となる軽量モデル(1B〜3B級)をローカルまたはステージング相当環境で動かし、実際のメモリ・CPU使用量を計測する
3. **合意**: 実測結果を片倉さんに報告し、本番投入するか・別マシンに切り替えるか(spec_v2.0.md 未決No.2とは別件、実行環境自体は確定済みだが実測結果次第で見直しうる)の合意を得てから次に進む
4. **抽象化**: Ollama呼び出しはサービス層(インターフェース)に隔離し、将来別マシン分離になってもコード変更なしで吸収できるようにする
5. **キュー化**: 一括登録処理は必ずqueue worker経由で実行し、Web側リクエストと資源を奪い合わない設計にする
6. **公演情報パイプライン実装**: 既存の`events.import`系(貼り付け→検証→人間確認→登録)のフローパターンを踏襲し、Ollama版を実装する
7. **`EventImportParser`の削除**: 6が実際に動作確認(検証データでの一括登録が成功)できてから削除する。**先に壊さない**。`EventImportParserTest`も同時に削除
8. **セットリストへの拡張**: 4章No.3(setlists設計)が片倉さんの確認を経て確定した後、公演情報と同一パイプラインを流用して着手する

各ステップの後に「何を確認すれば成功と判断できるか」(メモリ使用量が閾値以下、一括登録が期待通りJSON化される、等)を明示してから次に進むこと。
