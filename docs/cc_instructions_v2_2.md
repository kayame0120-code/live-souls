## 0. この指示書の読み方

- **本書に書いていない仕様を独自決定することを禁止する。** 迷ったら実装せず `QUESTIONS.md` へ隔離する。
- タスクは **T0 → T1 → T2 → T3 → T4 → T5** の順で実施する。T0は調査のみで、コードを変更しない。
- 各タスクの「受け入れ条件」は**機械検証可能な形**で書いてある。テスト・コマンド出力で証明できないものを「完了」と報告しない。
- **「検証済み」と書く前に、実際のコマンド出力を `REPORT.md` に貼ること。** 主張だけの完了報告は差し戻す。
- spec とCLAUDE.md が矛盾した場合、混ぜずに `QUESTIONS.md` へ隔離し、安全側で仮置きする。

## T1. JSONアップロード窓口の統合（最優先）

### 目的
現状、公演(events)とセットリスト(setlist)のJSON登録経路が分かれており、**JSONアップロード窓口にセットリストJSONを投げると弾かれる**。片倉さんの主運用（Claudeチャットで生成したJSONを手動アップロード）が成立しないため、これを最優先で直す。

### 位置づけ（重要・設計思想）
```
[主ルート・ゼロ円]  Claudeチャット → JSONファイル生成 → アプリのJSONアップロード窓口 → 確認画面 → 登録
[サブルート・従量]  アプリ内AI(OpenAI) → 画像/テキスト解析 → 確認画面 → 登録
```
- **JSONアップロード窓口が主**。片倉さん本人の運用はこちら
- **アプリ内AIはサブ**。招待メンバー（Claudeを使わない人）向け、および片倉さんの更新が追いつかない時の補助
- 両者は**同じ確認画面・同じ登録処理に合流する**。登録ロジックを二重に持たせない

### 変更内容
1. **入口を1つに統合する**：現在の「公演登録のJSONフォーム」を、**events / setlist の両方を受け付ける単一の窓口**にする
   - 1回のアップロードで **events JSON と setlist JSON を同時に複数ファイル投入できる**ようにする（一括挿入）
   - どちらの種別かはファイル内容（スキーマ）から判定する。ユーザーに種別を選ばせない
   - 種別判定できないファイルは、その1件だけを「不明」として確認画面上でスキップ扱いにする（全体を落とさない）
2. **確認画面を必ず経由する**：`アップロード → 確認画面(編集可) → 登録`
3. **セットリスト画面側の入口は残す**（公演詳細 → セットリスト画面からの登録動線は維持する）が、**登録処理は統合後の共通処理を呼ぶ**こと

### 禁止事項（違反したら差し戻す）
- **直INSERT を禁止する。** アップロードされたJSONを確認画面を経ずにDBへ書き込む実装を作ってはならない。AI経路・JSON経路・手入力経路のいずれも、**必ず人間の確認画面を挟む**
- 確認画面のためにDBの一時テーブルを作ることを禁止する（spec 3章：ステートレス方式。パース結果を編集可能フォームへ渡し、送信時に直接INSERTする既存 `events.import.parse` → `events.import.store` と同型）
- setlistの登録ロジックを、events用とは別に新規実装することを禁止する（合流させる）

### バリデーション（T0の調査結果を踏まえて実装）
- サーバー側で**JSONスキーマ検証を必須とする**。クライアント側の検証だけに依存しない
- 検証対象：必須キーの存在・型・`sort_order`等の整数性・FK参照先の存在（event_idが実在するか）
- **不正なJSONで例外・500を出してはならない。** 日本語のエラーメッセージを画面に表示し、どのファイルの何行目・どのキーが問題かを示す
- **不足データを勝手に補完することを禁止する**（例：event_dateが無いJSONを「今日」で埋める等）。不足は不足としてエラー表示する

### 受け入れ条件
- events JSON と setlist JSON を**同時にアップロード**して、両方が1つの確認画面に並び、送信で両方登録できる（Featureテストで証明）
- setlist JSON 単体でも、統合窓口から登録できる（Featureテストで証明）
- 不正JSON（必須キー欠落／型不一致／存在しないevent_id参照）の3ケースで、**500ではなく日本語エラーが返る**（Featureテストで証明）
- 確認画面を経由せずに登録できる経路が存在しないこと（ルート一覧を `REPORT.md` に貼って証明）
- 既存テストが全て通過

## T2. AI一括登録の改修（画像主・テキスト従／OpenAI一本化）

spec 4.1 に準拠。

### 変更内容
1. **入力を「画像主・テキスト従」にする**
   - 画像アップロードを画面の主役に置く。テキスト貼り付け欄は**折りたたみ**で残す
   - 画像：**1回あたり最大5枚**、形式は **jpeg / png / webp**（heicは対象外・既存の写真受付と同じ制限）
   - 解析用画像は**永続保存しない**。解析完了後に一時ファイルを削除する。`attendance_photos` のストレージ(Tigris)には載せない
   - 対象は3系統すべて：公演 / セットリスト / 当落締切
2. **LLMドライバをOpenAI一本にする**
   - `LLM_DRIVER=openai` のみ実装する。**Ollamaドライバは実装しない**（片倉実測で出力途中切れ・低速のため不成立）
   - `LlmService` インターフェースは**維持する**（テスト用フェイクドライバのために必要）
   - **テストで実APIを叩くことを禁止する。** フェイクドライバを用意し、テストはそれを使う
   - ビジョン対応モデルを使う。モデル名は `.env` または設定ファイルで差し替え可能にする
3. **確認画面にツアーの所属グループ選択を含める**（公演系統のみ・T3と連動）

### 禁止事項
- **E2E対象フィールド（`member_no` / `login_id` / `password`）の平文を LlmService に渡すことを禁止する**（セキュリティ基準No.11）
- LLM出力が不正・不完全な場合に、**勝手に補完して登録しないこと**。日本語エラーを表示し再実行を促す

### 受け入れ条件
- 画像5枚を投入して確認画面まで到達するFeatureテスト（フェイクドライバ使用）が通る
- 6枚目を拒否するテストが通る
- 解析後に一時ファイルが残っていないことを確認するテストが通る
- Ollama関連のコード・設定が残っていない（`grep -ri ollama` の出力を `REPORT.md` に貼る）

## T3. 公演タブのグループ束ね（3階層化）

spec 2.2 / 3章 / 3.3 / 5章に準拠。

### 変更内容
1. **マイグレーション**：`tours.idol_group_id`（FK → `idol_groups`、**nullable**）を追加。既存行は `null` のままでよい
2. **公演タブを3階層にする**：
   ```
   ①グループカード一覧 → ②そのグループのツアー一覧 → ③公演詳細(配下events)
   ```
   - ①：ツアーが1件以上紐づく `idol_groups` を**全てカード表示**。`idol_group_id` が null のツアーがあれば「未分類」カードを出す
   - ②：「＋公演を追加」「一括登録」の入口は**この階層**に置く（グループ文脈を引き継いで登録できるように）
   - ③：ツアー編集から所属グループを後付け・変更できる
3. **グループ追加UI**：`idol_groups` にユーザーが画面から行を追加できる入口を作る
   - **重複チェック必須**：前後空白を除去した上で既存名と照合し、一致すれば新規作成せず既存を選ばせる（共有マスタの表記ゆれ分裂を防ぐ）
4. **当落タブは変更しない**（2階層のまま）

### 禁止事項
- `idol_groups` の削除UIを作らないこと（共有マスタであり他ユーザーが参照しうるため）
- 既存ツアーへの `idol_group_id` をマイグレーションで機械的に一括紐付けしないこと（誤紐付けのリスク。null のまま手動で紐付ける）

### 受け入れ条件
- 3階層の遷移がFeatureテストで通る
- グループ追加で「なにわ男子」と「なにわ男子 」（末尾空白）が別行にならないテストが通る
- `idol_group_id` が null のツアーが「未分類」カードに出るテストが通る

## T4. 操作性の改修（スワイプ／日本語化）

spec 0章 / 2.2 / 5章に準拠。

1. **バリデーション日本語化**：`APP_LOCALE=ja`。バリデーションメッセージ・属性名（会員番号・入会日・開演時間 等）を日本語化する。導入方法（公式langファイル／`laravel-lang`等）はCC裁量。**英語メッセージが1つでもユーザーに露出したら不合格**
2. **タブのスワイプ切替**：メイン5画面（ホーム/参戦記録/名義/当落/公演）間を左右スワイプで移動可能にする（タップと併存）
   - **詳細・編集・複製・AI一括登録の各画面ではスワイプを無効にする**（誤操作防止）
   - 画面内の横スクロールUI（navピル・FCタブ・フィルタchips）との競合処理（角度閾値等）はCC裁量

### 受け入れ条件
- 主要フォーム（名義登録・公演登録・参戦記録）で意図的にバリデーションエラーを起こし、**表示された日本語メッセージのスクリーンショットまたはテスト出力**を `REPORT.md` に貼る
- スワイプ無効画面のリストが実装と一致していることを `REPORT.md` に明記

## T5. セキュリティ改修（E2E化ほか）

**要件・合否基準の正本は `security_requirements_v1_1.md` / `security_criteria_v1_1.md`。本書では重複定義しない。**

### 進め方
1. 実装前に、**鍵階層の図**と**E2E対象フィールドが通る全経路の一覧**を `REPORT.md` に提出する（基準No.9・No.11は画面を見ても判定できないため）
2. その後に実装する
3. 実装後、`security_criteria_v1_1.md` の**全16項目**に対してYES/NOで自己判定し、根拠（テスト・コード箇所）とともに `REPORT.md` に表を出す

### 特に落としやすい2点（基準ファイル「検品時の注意」参照）
- **No.1 と No.11 は別物**：「サーバーから平文を取り出せない」だけでは不十分。「サーバーに平文が届いていない」こと。フォームで平文を送信しサーバー側で暗号化する実装は不合格
- **No.4 と No.9 は別物**：パスワード**リセット**（旧PW不明）ではデータ復元不可、パスワード**変更**（旧PW既知）ではデータを失わない。両方をテストで確認すること

### 受け入れ条件
- 基準16項目の自己判定表が根拠つきで提出されている
- 満たせない項目は独自解釈で進めず `QUESTIONS.md` へ隔離されている
