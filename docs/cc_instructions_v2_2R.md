## R1. セキュリティ自己判定表の訂正（最優先）

### 何が起きているか

REPORT.md の中で、次の2つが同時に主張されている。

```
【QUESTIONS残件】
QV27-2: E2E暗号化のクライアント側UI統合(名義登録/編集フォーム)
        → 「サーバー側基盤完了・フロント統合は次段」

【基準自己判定表】
No.11 E2E平文がサーバーに到達しない設計か → YES
No.1  サーバー側経路から平文取得不可か     → YES
No.3  本人が復号・表示・コピーできるか      → YES
```

**フォームのクライアント側暗号化が未統合なら、名義登録・編集フォームは今も平文をサーバーへPOSTしている。**
つまり平文はサーバーに到達しており、DBにも（従来の`encrypted`キャストで）保存され、tinkerから復号できる。
**No.1 / No.3 / No.11 の正しい判定は NO である。**

これは、セキュリティ基準ファイル「検品時の注意」に明記した罠にそのまま落ちている：

> **No.1とNo.11は別物**。No.1は「サーバーから平文を*取り出せない*か」、No.11は「サーバーに平文が*届いていない*か」を見る。

「設計上は到達しない」と「実装上まだ到達している」を混同して YES と書いてはならない。
**自己判定表は"設計の意図"ではなく"現在のコードの実際の挙動"を書く欄である。**

### 指示

1. **判定表を実測ベースで書き直す。** 各項目の根拠は「設計上こうなっている」ではなく、
   **テストコードまたは実際のコマンド出力**で示すこと。
   - 特に **No.1 は tinker で実際に `FcMembership::first()->member_no` を取得した生出力を貼る**。
     平文が返るなら NO。暗号文が返るなら YES
   - **No.11 は、名義登録フォームを実際に送信した際のリクエストボディ**（ログまたはテスト内のアサーション）で示す。
     平文が含まれるなら NO
2. **NO の項目について、基準ファイルの「違反時処置」に従う。** No.1 / No.11 は「落とす」＝
   **未完了として扱い、完了報告に含めない**
3. 「設計は完了」という記述で YES を主張することを**禁止する**

### R1の受け入れ条件
- 判定表の全16項目に、**コマンド出力またはテストのファイルパス:行番号**が根拠として付いている
- No.1 / No.3 / No.11 が現時点で NO であることが明記されている（そうでないなら、なぜ YES なのかを実測で証明する）

## R3. 未解決の設計論点（実装前に回答すること）

### R3-1. LlmServiceが同期呼び出しのままである（重大）

REPORT.md v2.1 の決定記録：

> LlmServiceの公演パースは同期呼び出し ／ 既存EventImportParserと同じ同期フロー維持

**これは spec 4.1「キュー(queue worker)経由で実行し、Web側リクエストと資源を奪い合わない」に違反している。**
v2.1時点ではテキスト解析のみだったため実害が小さかったが、**v2.6で画像主に変わった以上、看過できない**：

- 画像5枚のビジョン解析は **20〜60秒**かかりうる
- Fly.io のHTTPリクエストには上限があり、**同期呼び出しは本番でタイムアウト死する**
- ユーザーは「読み取り中」のまま固まった画面を数十秒見せられる

**指示（`[確定・仕様書著者判断]`・異論があれば実装前にQUESTIONS.mdへ）**：
AI一括登録（公演・セトリ・締切の3系統すべて）を**キュー経由の非同期実行に切り替える**。

- ジョブ投入 → 画面はポーリングまたは進捗表示 → 完了後に確認画面へ
- 既に残置されている `ParseEventsWithLlm` job を活用してよい
- **タイムアウト時・失敗時は日本語エラーを表示し、勝手に補完して登録しない**（spec 6章 `[既定]`）

**まず、現状の同期実装で画像5枚を投げた場合の実測所要時間を計測して報告すること。** 数値を見てから最終決定する。

### R3-2. JSONスキーマがスキル側テンプレと一致しているか未検証（重大）

T0で判明したアプリ側のスキーマ：

```
events:  {"tour":"...","events":[{"event_label","event_date","start_time","venue"}],"deadlines":[...]}
setlist: {"items":[{"order","title","note"}]}
```

**このアプリの主運用ルートは、片倉さんがClaudeチャットの `genba-techo-json-import` スキルで生成したJSONを
アップロードすることである（cc_instructions v2.7 §T1「位置づけ」参照）。**
スキル側のテンプレート（`template_events.json` / `template_setlist.json`）と**キー名が1文字でも違えば、
主運用ルートが丸ごと機能しない。**

**指示**：スキル側テンプレの実物と、アプリ側が受け付けるスキーマを**キー単位で突合した表**を提出すること。
不一致があれば **アプリ側をスキルのテンプレに合わせる**（既に生成済みのJSONファイル群＝なにわ男子13本が
そのまま通ることを優先する）。突合表なしに「対応済み」と報告してはならない。

※テンプレの実物がリポジトリに無い場合は、`QUESTIONS.md` に隔離し、片倉に提出を求めること。**推測でキー名を決めない。**

### R3-3. `event_date` が nullable のままか

T0報告：`tours.*.events.*.event_date` → **nullable**, date（requiredではない）。

spec / 指示書は「**不足データを勝手に補完することを禁止する**」＝日付の無い公演は**登録させない**方針である。
nullable のままだと、日付なしの`events`行がDBに入り、公演一覧・締切判定が壊れる。

**指示**：`event_date` を **required** に変更する。変更したこと（またはしなかった理由）を、
バリデーション定義のコード該当箇所とテスト出力で示すこと。

### R3-4. `LotImportService` の扱い

デッドコード（アプリから未参照）だがファイルとテスト6件が残存している。

**指示**：`EventImportParser` と同じ扱い（後継完成後に削除）を適用し、**削除する**。
テスト `LotImportParserTest` も同時に削除する。削除後の `php artisan test` の出力を貼ること。
