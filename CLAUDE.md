# CC運用プロトコル（恒久ルール）— laravel-starter v2

## 本書の位置づけ
- 本書は**全アプリ共通の恒久ルール**。アプリを作るたびに編集しない
- アプリ固有のミッション・仕様は `docs/spec.md` に書く（本書には書かない）
- 本書と、spec.md・cc_instructions・人間のチャット指示のいずれかに矛盾を発見した場合、
  自己解決して混ぜることを禁止する。`QUESTIONS.md` に隔離し、
  **安全側（実装しない/影響の小さい側）で仮置き**して続行する
- **隔離した事実は、そのタスクの報告（REPORT.md または差し戻し回答）の冒頭に必ず明記する。**
  黙って安全側に倒して終えることを禁止する（隔離自体は正しくても、報告しなければ
  人間には「指示が無視された」ようにしか見えず、原因調査のコストが跳ね上がる）

## 技術前提
- Laravel 12 / PHP 8.2
- ローカルDB: SQLite（スケルトン既定のまま）
- 本番: Fly.io（nrt）+ Fly Postgres。接続は `DATABASE_URL`（Flyが注入）一本
- `.env` はCCが作成・変更しない（初期セットアップは人間がREADMEに従い実施済みの前提。
  未セットアップの形跡があれば QUESTIONS.md へ記録して停止）

## 文書の優先順位
1. `docs/spec.md`（アプリ固有仕様）
2. 本書（運用ルール）
3. その他の docs/

## コマンド権限（3区分）
| 区分 | 対象 | CCの扱い |
|---|---|---|
| 自由実行 | ローカルの全コマンド／本番の**読み取り専用**コマンド（`fly logs` / `fly status` / `fly ssh console` での参照系 / `pg_dump` / `ls` / `fly sftp get`） | 承認不要。検証・予行のため**むしろ積極的に実行する** |
| 承認制 | `fly deploy` / `git push` / `gh pr merge` | 実装 → 検証 → 報告 → **人間の承認** → 実行 |
| 禁止 | 本番への**書込み系**直接操作（本番での `migrate` 手動実行 / `seed` / `pg_restore` / `migrate:fresh` / ファイル削除等）、`.env` の作成・変更 | 実行しない。必要な場合は runbook に記載し人間が実行 |

- 判断に迷うコマンドは「本番のデータ・設定を変えるか？」で判定する。変えないなら自由実行
- 本番から取得したダンプ・データは個人情報を含む。検証完了の報告後に削除し、削除も報告する

## 自律ループ（このサイクル以外の進め方を禁止）
```
[1] 計画   spec全読 → 実装タスクを依存順にTODO化 → PLAN.md に書き出す
[2] 実装   タスク単位で実装 → 都度コミット（小さく・日本語1行要約）
[3] 検証   下記「検証ライン」を全て機械照合する
[4] 修正   失敗時は必ず エラーログ取得（証拠）→ 原因特定 → 修正 → 再検証
           - ログを見ずに「よくある原因」から直すことを禁止
           - 同一箇所で3回失敗したら QUESTIONS.md に記録してスキップ、他タスク続行
           - 続行可能なタスクが尽きたら停止し、人間に報告（無限ループ禁止）
[5] 完了   検証ライン全通過 → REPORT.md 作成 → 終了
```

## 検証ライン（合格条件。全てYESで [5] へ）
**判定原則: YES は「exit 0」ではなく「出力内容の該当行を引用できること」で成立する。
引用のない YES は未回答として扱われる。**

| # | 判定条件 | 照合方法と証拠 |
|---|---|---|
| V1 | マイグレーションが**実際に実行されて**エラーなく通る | `php artisan migrate --force`。**タスクが新規マイグレーションを含む場合、対象マイグレーション名が Running/Done として出力された行を引用する。`Nothing to migrate.` は不合格**（適用済みDBへの空振り。`migrate:fresh` 等で未適用状態を作り直して再実行する） |
| V2 | 起動後 `/up` が HTTP 200 | 下記ワンライナー。`200` の出力行を引用 |
| V3 | テストが全通過（テストが存在する場合のみ） | `php artisan test`。**passed 件数の行を引用**（exit 0 のみの報告は不合格） |
| V4 | データ変換マイグレーション（既存行の書き換え）を含む場合のみ: **変換前の状態を再現した上での実行ログ**と**冪等性**（2回実行して二重変換しない）をテストまたは実行ログで証明 | 変換前カウント → 実行 → 変換後カウント → 再実行 → 無変化、の一連のログを引用 |

V2 照合ワンライナー（起動→確認→必ず後始末）:
```bash
php artisan serve --port=8123 &
SERVER_PID=$!; sleep 3
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8123/up
kill $SERVER_PID
```

## 裁量境界
| 区分 | 例 | CCの扱い |
|---|---|---|
| 仕様（挙動・データ・画面に現れるもの） | テーブル定義 / URL / バリデーション / 画面要素 | spec通りのみ。spec外は実装禁止 → QUESTIONS.md |
| 既定（specバッジ「既定」） | specが既定と明示した振る舞い | そのまま実装。変えたい場合は QUESTIONS.md（勝手に変えない） |
| 実装手段（外から見えないもの） | クラス分割 / セッション管理 / 変数名 / テスト構成 | **CC裁量**。ただし主要な決定は REPORT.md に記録 |
| 未決（specに「未決」明記） | — | 実装禁止。プレースホルダー指示があるものだけ従う |

## 絶対禁止
- `.env` の作成・変更
- 本番への書込み系直接操作（コマンド権限表の「禁止」区分参照）
- DBのENUM型（string + アプリ側バリデーションで代替）
- SQLite（ローカル）とPostgreSQL（本番）で挙動が割れる書き方（DB固有関数・生SQL依存）
- 1ファイルへのロジック集中（ControllerはServiceへ切り出す）
- specにない仕様の独自決定 / QUESTIONS.md への記録なしのスキップ
- **QUESTIONS.md へ隔離したことを報告に書かずに終えること**
- main への直接コミット。mainから `feature/xxx` を切って作業し、
  **mainへのマージとリモートへのpushは人間の最終確認後**（CCはローカルコミットのみ）

### 秘匿情報の出力禁止（恒久）
- REPORT・チャット・提示コマンドの出力に、パスワード・DB接続文字列・APIキー・トークンを平文で含めない
- 検証コマンドは出力がブール値・件数になる形（grep -c / test / diff）で設計する。
  秘匿値そのものの表示による確認は禁止（存在・形式の判定で代替する）
- 片倉に実行を依頼するコマンドも同基準。秘匿値の受け渡しが必要な場合はファイル経由
  （umask 077・使用後rm）とし、画面表示・転記をさせない
- 違反した場合、露出した秘匿情報のローテーションは違反者側のタスク一式として扱う
  （検証・確認コマンドまで含めて自己完結させる）

## 品質規約
- コメントは日本語
- specに算出式の境界値・エッジケースの記載がある場合は**必ずテスト化**する
  （テストが仕様の写しになっていることが検品の根拠になる）
- 複数アプリで再利用する見込みのあるドメインロジックは `packages/` 配下に切り出す
  （判定基準:「別テーマのアプリを作る時にそのまま require できるか」）

## デプロイ境界
- 本番の**書込み系**コマンド（fly deploy / 本番migrate / seed / restore）は人間が実行する。
  本番の**読み取り専用**コマンドはCCが自由に実行してよい（コマンド権限表参照）
- **fly.toml の `release_command` に `php artisan migrate --force` を設定する（標準）。**
  デプロイとマイグレーションの時間差は、データ形式を変えるデプロイにおいて
  「新コードが旧形式データを読んで例外になる窓」を生むことが実証されたため
  （現場手帖デプロイA・DecryptException窓）。`fly deploy` 自体が承認制である以上、
  release_command 経由のマイグレーションも人間の承認下にあり、統制は維持される
  ※ v1の「release_commandを仕込まない」規定はこれを撤回する
- 本番反映が必要な変更を含む場合、CCは `docs/deploy_runbook.md` を
  **人間が上から実行するだけの状態**に完成させてから終了する

## deploy_runbook.md の合格条件
- 記載する全コマンドは**実行実績のあるものだけ**とする：
  - ローカルで完結するコマンド → ローカルでの実行ログを添付
  - 本番の読み取り専用コマンド（バックアップ取得等）→ **runbook作成時点で本番に対して予行**し、実際の出力（ダンプサイズ等）を添付
  - 本番の書込み系コマンド（deploy等）→ 実行はできないため、構成要素（release_command の内容等）の根拠を記載
- バックアップ手順には**リストアの実演ログ**（ローカルの使い捨てDBに対して）を含める。
  「取れた」ではなく「戻せた」までが検証
- ロールバックの主経路は **pg_restore による復元**とする（マイグレーションの down() は
  旧イメージに存在しない・実行順の罠があるため、主経路にしない）
- アプリ名・URL・パス等のプロジェクト固有値は、**リポジトリ内の他プロジェクト由来の
  値が混入していないか grep で照合**し、結果を添付する（実例: 別アプリのURLがrunbookに
  2版連続で混入した事故）
- runbook提出時、前回差し戻しがある場合は**指摘→対応箇所→未対応理由の突合表**を添付する
